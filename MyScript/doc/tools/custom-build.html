Creating custom resources

  • Tools
  • Creating custom resources

    • You can build custom resources through our standalone interface (MyScript Language Resource Tool), through the API or through command lines. However, we strongly recommend that developers use the interface to build and manage their linguistic and subset knowledge resources, instead of using the command line tools.

    MyScript Language Resource Tool

    MyScript Language Resource Tool is an easy-to-use standalone interface for creating custom resource files. If you are not currently using it (and you are eligible to use it), you can obtain a copy via the SDK download site and install it. It is protected by a security license. You need to activate it with your security code to use it on a permanent basis.

    With this tool, you will be able to:

    • build your own resource files,

    • compile your resource files into .res,

    • manage these resource files.

    Here is a snapshot of the MyScript Language Resource Tool interface:

    MyScript Language Resource Tool makes use of four different applications for you to create custom resource files. This interface prevent you from using these four applications via command lines. It is provided with a complete documentation and video tutorials, once you have installed it. For this reason, the present documentation will not elaborate on the detailed use of MyScript Language Resource Tool. Instead, the present documentation will develop how to use the different applications that MyScript Language Resource tool makes use of.

    You will find out how to perform the different management tasks with each of the supported resource types in the next subsections. The four different tools that enable the creation of custom resource files, depending on the resource type to build are:

    Resource type Tool
    Lexicon MkLex
    Lexical units, regular expressions MkLU
    Subset Knowledge MkSk
    Archive MkArchive
    You must use the custom resource tools with --charset=UTF-8 to process CJK (Chinese, Japanese, Korean) characters, to make your own subsets, archives, lexicons or regular expressions.

    MkLex for lexicons

    MkLex is designed for the creation of lexicons. Standalone MyScript Language Resource Tool makes use of MkLex application for building lexicons. You can also use it directly, via command lines. MkLex will compile your custom list of words/expressions from text format into a binary *.res file format. It can be found in the rdk/tools folder in the MyScript root directory, in the bin folder corresponding to your system.

    Setting MkLex

    1. Execute the rdkConsole.bat file that is in the rdk folder to set the environment,

    2. The tool has options that you can set. To see the options, type the help command:

       % MkLex --help

    3. Follow the instructions.

    Compiling a text file into a *.res file

    1. Execute the rdkConsole.bat file that is in the rdk folder,

    2. Give your text file a .lex extension,

    3. In the console, go to the directory required and use this command line:

       % MkLex mywords.lex

    This will create a mywords.res file in the same location.

    Renaming a file

    1. Execute the rdkConsole.bat file that is in the rdk folder,

    2. Use the -o option:

       MkLex -o my_nice_lexicon.res mywords.lex

    Some text editors (for example, Notepad for Windows) add 3 bytes (a BOM or byte order mark) at the beginning of files to specify byte order when saving using UTF-8 encoding. Those bytes are not defined in the UTF-8 encoding standard. MyScript uses strictly standard UTF-8, so please make sure that you remove them before processing. Check your editor: some auto-detect or auto-edit BOMs.

    Adding content to an existing lexicon resource

    You cannot add words/expressions to any of the existing lexicon resources pre-defined in MyScript. For instance, it is not possible to add words to [language+country_prefix]-lk-text.res, [language+country_prefix]-lk-city.res etc. You can only modify lexicons that you have created yourself.

    Here is a summary of how to add words or expressions to an existing lexicon with the API:

    • Load the required lexicon object

    • Decompile

    • Add words to the lexicon

    • Compile it again

    • Store the resource you have created or use it directly

    MkLU for lexical units

    MkLU is designed for the creation of lexical units based on regular expressions. Standalone MyScript Language Resource Tool makes use of MkLU for building lexical units. You can also use it directly to compile a list of regular expressions in *.ludef format into a binary *.res file format. MkLU is located in the rdk/tools folder in the MyScript root directory, in the bin folder corresponding to your system.

    This subsection will discuss the regular expressions and their editing, but the impatient can skip these explanations and go directly to the Using MkLU section.

    The data format of the information you wish to recognize may be simply textual (word, phrase). Or, it may match another syntax that contains other elements to convey meaningful information. This is true for telephone numbers, dates, and e-mail addresses, among others. These types of elements are known as lexical units, and these are usually best recognized by lexical unit resources based on regular expressions.

    Editing regular expressions

    A regular expression is a series of characters and operators that describe or match character strings following a certain syntax (for instance, telephone numbers, dates, and e-mail addresses, among others). For MyScript purposes, an expression is written in a text file with operators. It must follow a strict grammar. Examples of this grammar appear below, and a more complete syntax in BNF form.

    MyScript uses regular expressions in:

    • .ludef files
      which contains one regular expression that fills many lines.

    • .regex files
      which is a list of regular expressions (and other text if necessary) expressing several variables.

    Regular expression operators

    A regular expression describes an item without making an explicit and exhaustive list of all its possible elements. For instance, it may describe a lexical unit, such as a date. Instead of listing all the existing dates, the regular expression allows describing the general format of a date.

    You can include any of the characters, digits or symbols from your active charset as characters in your expression.

    Below are the most common operators.

    Classical mapped character Description Example
    . Dot: matches any single character (except a newline character) .123 matches any character followed by 123 (e.g. a123, z123, 5123 and so on)
    [ ] Brackets, set delimiter: matches any of the characters contained within the brackets. [abc] matches a or b or c. [123] matches 1 or 2 or 3.
    [\^] Caret, negative set indicator: matches any symbol except those listed in the set [^xyz] matches any character except x or y or z
    - Dash, set range indicator: matches all the characters included in the given range. Be careful to use this only for simple ranges, as it takes into account their order in their charset, which does not always correspond to extended alphabets! It works with English upper and lower case letters and digits. It can be literal provided it is the first or last character in a set e.g [-123] [a-z] matches any lowercase English letter. [r-u] matches r, s, t or u. Literal usage: [-12] matches - or 1 or 2
    | Vertical bar. Is a separator that defines alternatives saint|st matches saint or st
    \ Operator modifier: considers that the character following it has special meaning, e.g. that a potential operator is a character and not an operator \. is used to insert the dot symbol in a regular expression. n is the character n but \n matches a linefeed/newline operator (NB: only in the API, not in an expression for MkLU)
    + Plus sign: quantifier. Defines that the character or expression preceding it must be present at least once. aa+gh matches aagh, aaagh, aaaagh and so on, but does not match agh.(ab)+ matches ab, abab, ababab and so on but not aba.
    * Asterisk: quantifier. Defines that the character or expression preceding it can be present 0 times, once, or many times. 0*99 matches 99, 099, 0099 and so on. a(bc)* matches a, abc, abcbc and so on but not abcb.
    ? Question mark: quantifier. Defines that the character or expression preceding it may be present 0 times, or once. Mon(day|\.)? matches Mon, Monday and Mon.(La )?Roda matches Roda or La Roda
    ( ) Parentheses: matches whatever expression is inside it, indicates the start and end of a group. Jan(uary)? matches Jan and January
    {exp_identifier} Inserts the definition of the regular expression defined under the name exp_identifier, which has been specified earlier in the file day_of_birth = {num_day} refers to {num_day}= 0[1-9] | [1-2] [0-9]|3[0-1] that was defined in another line of the same expression.
    {#n} Inserts the marker n from 0 to 8 (e.g. marker 0 is {#0}). A marker is used to tag a part of a language definition (ludef) in order to localize it in the recognition result, without influencing the rest of the recognition. Markers fall within the reserved range, namely U+F0010 to U+F0018. ({#0} this is our first alternative | {#1} this is our second alternative). When {#1} appears in the recognition result, you will understand better what the recognizer has processed.
    $ Matches the empty string at an end of line. It can be used to constrain the recognizer to force a specific part of the language to coincide with the end of a line. Use it in conjunction with an @enable end-of-line directive. @enable end-of-line, or @enable start-of-line, or %[0-9]+$, defines a string made entirely of digits (% and $ represent the beginning and end of a line).

    Be aware that the MyScript engine cannot handle any regular expression ended by a non observable character (spacing characters, markers and the NUL character).

    Ludef objects

    Below you will find the main objects you need to know to work with the *.ludef BNF1 Grammar: the .ludef file is the first step in making a lexical unit resource file based on a regular expression. You can see a full grammar set out in BNF style in a dedicated section.

    Object Description Syntax
    ludef A .ludef file is the way to define lexical units with regular expressions. The file is made up of a sequence of lines.  
    line A line is either a comment line or an empty line or a directive or an instruction.  
    comment line A line with a hash (#) as the first non-whitespace character, followed by a sequence of any characters.  
    whitespace An empty sequence of spaces (U+32) and/or tab characters (U+9). Expressed as a space or the \t operator for a tab  
    directive These are for controlling the compiler; elements have to be separated with whitespace. @set charset charset_name or @set frequency-scale scale_value or v@enable optionor@disable option“
    import The import command imports external objects into the .ludef file (see below for details). @import file_type file_name
    string-list A list of strings from another file; this can be a text file with a list of names or postcodes, for example. Used in commands such as @import string-list mynames.txt
    regex-list A list of regular expressions from another file; this can also be a lexicon that uses regular expressions to express certain variations. Differs from a .ludef in that the .ludef is a large regular expression, that can even import other items. You can however use frequency values in a regex-list, using a particular method. To use a regex-list, you must import it into a ludef file before compiling it using the MkLU custom lexical unit tool. Used in commands such as @import regex-list mycities.regex Its contents may contain standard words or names in addition to items expressed with operators; imagine that mycities.regex contains items such as:S(AIN)?T HARGROVE, (ISABA\|IZABA), (LA )?SERNA
    {NUL} This operator is strictly limited to MyScript expressions. {NUL} inserts a NUL character in the regular expression, which can be used for various reasons. The NUL character is recognized anywhere without consuming any input i.e. it has no effect on the actual recognition. When you attach a frequency to it, that frequency can be accumulated with the other frequencies on the recognition path. Before you can use this expression, you must import it into the ludef file in question, using this command line: @import ludef “defNUL.ludef”. The defNUL.ludef file can be created automatically by right-clicking the sources folder in MyScript Language Resource Tool and choosing defNUL.ludef. The same action can be carried out for the defSURROUNDING.ludef file, required when creating circled character recognition.

    Importing objects into a .ludef

    You can use the import command to import the following into a .ludef file:

    • another ludef file (ludef type),

    • a list of strings from another file (string-list type),

    • a list of regular expressions from another file (regex-list type),

    • another compiled .res resource (resource type).

    The command syntax is:

    expression = @import file_type "file_name"

    The types you can use for file_type are ludef, string-list, regex-list or resource.

    Here is an example of its syntax:

    firstname = @import string-list "firstname.lex"

    Casing in lexical unit resources

    Casing does affect your recognition results. Casing is important for lexicons, as the case used in lexicon entries can have a direct consequence on how the recognizer recognizes. Vocabulary properties allow you to control more precisely the effect of the casing on the recognition result, in MyScript Text. This is described in detail in the help files for MyScript Language Resource Tool, used for building custom resources.

    Examples of regular expression syntax

    Example 1
    Some regular expressions can be fairly simple. For example, the following regular expression is for an age data format. It takes into account ages from 24 to 80:

    2[4-9] | [3-7][0-9] | 80

    Example 2
    Here is another example of a regular expression that will describe an id_number object. Each line is commented, so you can understand how the file is constructed:

    num_month = 0[1-9]|1[0-2]

    This defines the expression num_month, which consists of a two-digit number from one of two groups: either a number from 01 to 09 (0[1-9]) or (symbolised by |) a number from 10 to 12 (1[0-2])

    num_day = 0[1-9]|[1-2][0-9]|3[0-1]

    This defines the expression num_day which consists of a two-digit number from one of three groups: either a number from 01 to 09 (0[1-9]) or (symbolised by |) a number from 10 to 29 ([1-2] [0-9]) or (\|) a number from 30 to 31 (3[0-1])

    day_of_birth = {num_day}

    This says that the element day_of_birth corresponds to the num_day element described earlier.

    month_of_birth = {num_month}

    This says that the element month_of_birth corresponds to the num_month element described earlier.

    year_of_birth = [0-9][0-9]

    This describes a two-digit number between 00 and 99.

    extra_number = [0-9]

    This describes a single digit, which is a check digit for our id_number

    sep = [-/]?

    This describes potential separators in the expression, namely a dash or a slash. The ? signifies that each separator may appear once, or it may be omitted.

    id_number = {day_of_birth}{month_of_birth}{year_of_birth}{sep}{extra_number}

    This assembles all the composing parts of the expression that have been defined elsewhere within it. You can see in which order the various parts must appear.

    {id_number}

    This is the identifier that stands for the entire expression. This is used to identify the expression during compilation.

    Example 3
    The following is another example of a more complex data format. Here is an abstract representation of the format we are seeking to recognize:

    The complex reference number above is divided into several sections, namely:

    • area code: a string list called area_code.txt, where area codes are entered line by line

    • separator: this could be a dash (-) or a backslash (\)

    • alphanumeric_number: a compound of @1, which is an uppercase character between A and D, @2, which is either the letter “f”, “u” or “n” and #1 which is a digit between 0 and 9.

    • number: #2 is a digit between 0 and 999

    Using the rules that we specified concerning regular expression syntax, here is an expression that defines the data format above:

    area_code = @import string-list "area_code.txt"
separator = - | \\
alphanumeric_number = [A-D] [fun] [0-9]
number = [0-9] [0-9]? [0-9]?
reference_number = {area_code} {separator} {alphanumeric_number} {separator} {number}
{reference number}

    Regular expression BNF grammar

    Here is a syntax in BNF1 grammar form of the elements you may happen to use when creating regular expressions in MyScript.

    	/* ----------------------------- First level syntax ----------------------- */

	/** A *.ludef file is a sequence of lines */
	ludef           ::= line
	                |   line '\n' ludef
	/** A line is either a comment line or an emtpy line or a directive or an
	    instruction */
	line            ::= comment-line
	                |   whitespace
	                |   directive
	                |   instruction

	/** A comment line is a line with a hash as the first non-whitespace character,
	    followed by a sequence of any character */
	comment-line    ::= whitespace '#' comment
	comment         ::=
	                |   any-char comment

	/** A whitespace can be a empty sequence of spaces (U+32) and/or
	    tabulations (U+9) characters */
	whitespace      ::=
	                |   ' ' whitespace
	                |   '\t' whitespace

	/** Directives are for controling the compiler */
	directive ::= whitespace '@set' whitespace 'charset'.whitespace charset-name whitespace
	          |   whitespace '@set' whitespace 'encoding'.whitespace encoding-name whitespace
	          |   whitespace '@set' whitespace 'frequency-scale'.whitespace scale-value whitespace
	          |   whitespace '@set' whitespace 'accentuation-scheme'.whitespace accentuation-scheme-value whitespac
	          |   whitespace '@set' whitespace 'casing-scheme'.whitespace casing-scheme-value whitespace
	          |   whitespace '@enable' whitespace option whitespace
	          |   whitespace '@disable' whitespace option whitespace

	/** Sets the charset */
	charset-name ::= ...          /* Name of a valid charset. 
	This must either be UTF-8 or an 8-bit charset like US-ASCII, ISO-8859-15 or 
	Windows-1252																*/
	                                 
	/** Sets the text encoding */
	encoding-name ::= '8bits'     /* Each character is 8 bits wide              */
	              |   'UTF-8'     /* UTF-8 encoding                             */

	/** Sets the frequency scale of the lexical unit resource */
	scale-value ::= floating-point-constant

	/** Sets the accentuation scheme */
	accentuation-scheme-value ::= 'lax'
	                          |   'strict'

	/** Sets the casing scheme */
	casing-scheme-value ::= 'lax'
	                    |   'legacy'
	                    |   'strict'

	/** Options that controls the regular expression syntax */
	option ::= 'start-of-line'    /* Interpret start-of-line operator (%)       */
	       |   'end-of-line'      /* Interpret end-of-line operator ($)         */

	/** An instruction is composed of a left and a right part */
	instruction     ::= left-part right-part

	/** The left part is optional. It contains an identifier and a frequency */
	left-part       ::=
	                |   whitespace ident whitespace frequency '='

	/* ----------------------------- Identifier (same as C) ------------------- */

	ident                   ::= ident-first ident-continue-sequence
	ident-first             ::= 'A' | 'B' | ... | 'Z'
	                        |   'a' | 'b' | ... | 'z'
	                        |   '_'
	ident-continue-sequence ::=
	                        |   ident-continue ident-continue-sequence
	ident-continue          ::= ident-first
	                        |   '0' | '1' | ... | '9'

	/* ----------------------------- Frequency -------------------------------- */

	/** The frequency is optional and is introduced by a slash and defined by
	    either a symbolic frequency constant: (low|medium|high)-frequency or a
	    floating-point constant between 0 (for low-frequency) and 100 (for high-
	    frequency). Please note that 0 to 100 is merely a scale, and the sum of all frequencies does not have
	    to add up to 100. Frequency is used to weight some alternatives in relation to others, to manage priorities.*/
	frequency       ::=
	                |   '/' whitespace frequency-value whitespace

	frequency-value ::= 'low-frequency'
	                |   'medium-frequency'
	                |   'high-frequency'
	                |   floating-point-constant

	right-part      ::= whitespace import whitespace
	                |   whitespace exp whitespace

	/* ----------------------------- Import command --------------------------- */

	/** Import command allows you to import:
	    - another ludef file,
	    - a list of strings from another file,
	    - a list of regular expressions from another file,
	    - a lexicon resource,
	    - another data format resource. */
	import          ::= '@import' ' ' file-type ' ' '"' file-name '"'
	file-type       ::= 'ludef'
	                |   'string-list'
	                |   'regex-list'
	                |   'resource'
	file-name       ::= ...       /* System dependent!                          */

	/* ----------------------------- Regular expression ----------------------- */

	exp             ::= '(' exp ')'
	                |   '[' set ']'
	                |   '{' ident '}'
	                |   exp '|' exp
	                |   exp '?'
	                |   exp '*'
	                |   exp '+'
	                |   exp exp
	                |   '.'
	                |   symb-but-op
	                |   '\' symb

	set             ::= '^' symb-list
	                |   symb-list

	symb-list       ::= symb-list-elt
	                |   symb-list-elt symb-list

	symb-list-elt   ::= symb-range
	                |   symb

	symb-range      ::= symb '-' symb

	/* ----------------------------- Character types -------------------------- */

	any-char        ::= ...       /* Any character but the new line '\n'        */
	symb-but-op     ::= ...       /* Any symbol but the reg. expr. operators    */
	symb            ::= ...       /* Any symbol                                 */

	/* ----------------------------- Floating-point constant ------------------ */

	floating-point-constant ::= integer-part
	                        |   integer-part decimal-part
	                        |   integer-part decimal-part
	                        |   decimal-part
	integer-part            ::= digit-sequence
	decimal-part            ::= '.' digit-sequence
	digit-sequence          ::= digit
	                        |   digit digit-sequence
	digit                   ::= '0' | '1' | ... | '9'

    Using frequency in regular expressions

    It is possible to give more or less importance to different definitions in a ludef. This is done by attributing a frequency value to certain elements. This indicates to the recognition engine how likely that item is to occur. The recognition engine will give weighted entries pre-eminence as opposed to other entries.

    You can also adjust the scale of those frequencies (frequency scale), to change the importance of the frequencies for a whole resource. The frequencies will still count but they may be attenuated. This is a way of weighting one resource against another.

    Frequency values

    Frequency values are added to a regular expression in order to:

    • notify the recognizer how much it should favor one expression over another during the recognition process. It is a way of ranking the potential input from the most common to the most unlikely.

    • add an out-of-lexicon (OOL) capability but as a last resort: by giving the OOL capability a low frequency, it will only be taken into account when other options fail to find a result.

    Frequency values are expressed in the form of a number from 0 to 100, 100 being the highest weighting. You can also use a set value of high-frequency, medium-frequency or low-frequency. They are added using this syntax:

    name_of_expression/frequency-value=definition

    Here is an example of giving precedence to different entries in the same ludef file:

    /*Letter uses the default value of 50. If we want to allow digits but emphasize that they are less likely to occur than letters in words, we need to attribute to the next entry a weaker frequency scale, for example of 10.*/
Letter = \[A-Z\]
Digit /10 = \[0-9\]
Word = ({Letter}|{Digit})+
{Word}

    And here is an example of a ludef using frequency values on imported resources:

    list1/80 = @import string-list "my_medical_terms.lex"
list2/20 = @import resource "en_GB-lk-standard_lexicon.res"
my_custom_weighted_resource={list1}|{list2}
{my_custom_weighted_resource}

    For exical Unit resources, the default frequency value is 50 and is used for each element. The frequency value is a number between 0 and 100, with 100 for highly probable elements and 0 for the least probable elements. The scale is linear.

    If you do not specify a frequency value, the default of 50 is taken as given. For example, the following ludef file:

    Letter = \[A-Z\]
Word = {Letter}+
{Word}

    has the same effect as if you had written it as:

    Letter /50 = \[A-Z\]
Word /50 = {Letter}+
{Word}

    Special remark concerning the use of frequency when creating lexical units “on the fly”
    In this case the frequency needs to be calculated differently. Use this formula to calculate the equivalency: (frequency - 50)/50. For example, taking the same example as given above, we would have this:

    Vocabulary list1 = (Vocabulary)EngineObject.Load(engine, "my_medical_terms.res");
Vocabulary list2 = (Vocabulary)EngineObject.Load(engine, "en_GB-lk-standard_lexicon.res");
lexicalUnit.ImportVocabulary("list1", list1, 0.6F);
lexicalUnit.ImportVocabulary("list2", list2, -0.4F);
lexicalUnit.DefineRegex("my_custom_weighted_resource","{list1}{list2}");
lexicalUnit.DefineMainRegex("{id_number}");

    Note that the original example uses a .txt file, but you must use a .res for this usage. Here, a .NET example is used, but the same principle applies to code in Java or C. Using frequency values with a regex list

    You can use frequency values with a regex list: you weight the values in the regex list and import it into a ludef file that defines the weightings, using the {NUL} expression. The defNUL.ludef file defines the named expression NUL as the NUL (zero) character, so you need to import that in order for NUL to be understood in the expression. Here is an example of a ludef file (called a.ludef) using frequencies:

    a.ludef
@import ludef "defNUL.ludef"
w1/low-frequency={NUL}
w2/medium-frequency=(NUL}
w3/high-frequency={NUL}
@import regex-list a.rl

    Here is the a.rl regex-list imported above (it weights “anne” and “anna” as being more probable than “anny” which has a lower frequency value):

    a.rl
ann{w3}e
ann{w2}y
ann{w3}a

    The advantage of using this method is that it does not cause strain on the linguistic path system used by the recognition engine (see the document on the recognition path bandwidth in order to understand this system).

    Frequency scales

    Frequency values are, naturally, linked to a scale. You can adjust this scale’s magnitude for more dramatic differences in weighting. To do this, use a @set frequency-scale command at the beginning of your ludef; for example, to multiply all the weightings by five, use:

    @set frequency-scale 5
list1/80 = @import string-list "my_medical_terms.lex"
list2/20 = @import resource "en_GB-lk-standard_lexicon.res"
my_custom_weighted_resource={list1}|{list2}
{my_custom_weighted_resource}

    To give you an idea of how this weights the entries, let’s look at how changing the scale affects the frequency:

    With no change to the scale:
    Frequency=50 means no added cost for the recognition of this entry,
    Frequency>50 means we promote this entry,
    Frequency<50 means this entry is penalized.

    The frequency-scale setting allows you to change all of the frequencies of one resource by changing its scale. For example, from a default scale of 1, you can go up to 2 to double promotion and penalization:

    With a frequency scale of 2:
    Frequency=50 will not be changed (no cost)
    Frequency>50 will be further promoted
    Frequency<50 will be further penalized

    Knowing which value/scale magnitude to use

    Here comes the bad news: you will find that you will need to experiment with several different weightings and scale magnitudes before you find the recipe that best suits your application. There is no hard and fast rule for this; it relies mainly on your experience with MyScript resources.

    Using frequency can be a complicated task for inexperienced users: if you wish, you can call upon MyScript’s on-demand services for help in setting up correct frequency values and scales.

    Extending resource coverage

    It is possible to add to a resource the capability to recognize items that are not described in a lexicon or lexical unit resource attached to a recognizer. Normally, if you attach a lexicon or lexical unit to a recognizer, this is a constraint. The recognizer will only produce results that match the words or expressions described in the attached resource(s). If you add an additional capability, you give the recognizer the chance to recognize words that are not included in the attached resource(s).

    When you add expressions to extend the resource coverage, you should give it low priority so it will come into action only if a match cannot be found with the main resource contents, for example the list of names in a name resource.

    Please note that some linguistic knowledge resources have been changed. The extended resource coverage is no longer added as a default, and an extra resource with “extended” in its name is delivered, with only that resource containing an extended coverage.

    When is it appropriate to extend resource coverage?

    The only situation in which we recommend that you do not extend your resource, is if you want to recognize only a limited set of words or expressions, and none other than these. In the majority of situations, however, you will find it useful to use some extension capability as you will want to allow for the possibility that you will get input that does not match the attached resources.

    When you add an extension capability to a resource, you will generally set a frequency value to it, with a lower frequency than the other resource contents, so that the extension expression will not have high priority during the recognition process. Frequency values set priority for resource items and are described in the previous section.

    How to extend your resource coverage

    Extending your resource coverage is relatively complex: finding the correct contents and weightings is largely a question of experience and testing. There is unfortunately no magic formula as the best compromise depends on many factors, including the true percentage of extended entries that the application will have to manage.

    You need to add expression to the attached ludef file: to this ludef file, you give a lower frequency than to the other linguistic resources you are using (so that the added expression does not have priority over them). Alternatively you can make a separate file, compile it as a resource and attach that resource to the recognizer along with the other linguistic resources.

    Here is an example. Start in an existing.ludef file, and define your extension expression at the end of the file, or create your expression in a new luedf file. For example:

    iso/20 = \[a-z0-9\]
ool = {iso}+
{ool}

    This capability can recognize any combination of upper or lower case letters and digits. This example assumes some knowledge of regular expressions.

    Next, build the resource in .res format using the MkLU tool or in the MyScript Language Resource Tool.

    Extending resource coverage in Asian languages

    This is a little more complex.

    1. Put the characters concerned in a lexicon file: for example, in a file called kanji_list.lex.

    2. Next, use this expression in your ludef or regex-list file to add coverage taking the new characters into account:

      kanji=@import string-list "kanji_list.lex"
ool/20={kanji}+
{ool}

    3. Build your resource with this source file.

    To add the capacity to recognize any character (providing it is available in the AK resource being used), use an expression of this type:

    ool/0=.+
{ool}

    Frequency in extended coverage expressions

    In order to create your own extension capabilities, you will need to be able to manipulate frequencies and have good experience with creating lexical unit resources.

    MkSK for SK

    MkSK is designed for the creation of subset knowledge (SK) resources. The interface MyScript Language Resource Tool uses this application, preventing you from using the command lines. However, you can use it directly via the command lines. MkSK is located in the rdk/tools folder in the MyScript root directory.

    Remember that SK resources are used to state which characters are valid for recognition and which are to be disabled (e.g. you can recognize only digits, only lowercase letters, letters and symbols but not digits, and so on). You can use the basic subsets we have provided or make your own, if you want to use a very limited set of characters (for example, number 1 to 5, a few important letters, etc.). Only the characters you have given in your subset will be enabled. Any others will not be recognized.

    The types of subset bases at your disposal for creating SK resources are listed on the page that describes subset knowledge resources. All the basic subsets, in the form of .sk resources (upper.sk, digit.sk and so on) are stored in rdk/src/name_of_language. Some languages have slightly different subsets, for example the East Asian languages such as Japanese which have subsets for hiragana, katakana etc.

    You can combine several subsets (SK files) into one resource (e.g. alpha+digit.res).

    Setting MkSK

    1. Execute the rdkConsole.bat file that is in the rdk folder.

    2. The tool has options that you can set. To see the options, type the help command:

       % MSK --help

    3. Follow the instructions.

    Compiling a *.sk file into a *.res file

    1. Execute the rdkConsole.bat file that is in the rdk folder.

    2. In the rdkConsole.bat console, go to the directory required, for example:

       % cd src/fr_FR

    3. Use this command line:

       % MkSK name_of_subset.sk

    This will create a name_of_subset.res file in the same location.

    Combining several *.sk files into a single *.res file

    1. Execute the rdkConsole.bat file that is in the rdk folder.

    2. In the rdkConsole.bat console, go to the directory required, for example: ~~~ % cd src/fr_FR ~~~

    3. Use this command line:

      % MkSK subset1.sk subset2.sk

    This will create a subset1+subset2.res file in the same location.

    Remember that you can attach only one SK resource to a recognizer. Attaching more than one will detach the previously attached resource.

    Special note regarding Czech recognition

    Czech language has a special character, ch, which is a digraph (two characters that are treated as a single letter). The API provides different methods to handle either the character “ch” or the characters “c” then “h”:

    Methods Description
    EnableCharacters("ch") it iterates through the input string and enables “c” then “h”
    EnableCharacter("ch") it treats “ch” as a whole, hence enabling the “ch” character specific to Czech.

    When making SK source files for Czech, use the following syntax:

    • Space or line breaks are delimiters: they delimit mode changes or inputs.

    • There are 2 modes: “@characters” and “@character”

    • “@characters” is the default mode (this also ensures that previous SK files keep working as they used to): every input between 2 delimiters is considered as a sequence of characters

    • “@character” changes the current mode: every input between 2 delimiters is considered as a whole character.

    Here is an example of this in an SK source file:

    abcdefghijklm
nopqrst
uvwxyz

@character ch CH

CH

@characters ABCD

EFG
H
IJKLMNOPQRST
UVWXY

    • MkCharacterSK starts by disabling all characters then sets @characters as the default current mode.

    • “abcdefghijklm” is treated as a single input parameter to “enablecharacters” and enables “a”, then “b”, then “c” … up to “m”. The line break is ignored.

    • “nopqrst” is treated as a single input parameter to “enablecharacters” - The line break is ignored.

    • “uvwxyz” is treated as a single input parameter to “enablecharacters” - The line break is ignored.

    • then, @character changes the current mode. the space is ignored.”ch” is treated as a single input parameter to “enablecharacter”. The space is ignored. “Ch” is treated as a single input parameter to “enablecharacter”. The line break is ignored.

    • “CH” is treated as a single input parameter to “enablecharacter”. The line break is ignored.

    • then, @characters changes the current mode. The space is ignored.

    • “ABCD” is treated as a single input parameter to “enablecharacters”, enabling “A”, then “B” etc. The line break is ignored. etc…

    As another example, the czech sk source file for lower case letters (lower.sk) is expressed like this, so “ch” is considered as a single input:

    @characters
aábccddeéefghiíjklmnnoópqrrsšttuúuvwxyýzž
@character
ch

    MkArchive for archives

    MkArchive is designed for the creation of archives. Standalone interface MyScript La,nguage Resource Tool uses this application for building your archive. You can also directly use it for the same purpose, via command line. It will compile your custom archive resource.

    The MkArchive tool is located in the rdk/tools folder in the MyScript root directory.

    Guidelines

    An archive resource can contain any of the usual types of resource (Alphabet, Linguistic, Subset). Archive resources must still follow the rules of use for these resource types, so generally speaking:

    • An archive resource can contain one or more AK resources (but this depends on whether you are using a structured or unstructured recognizer).

    • An archive resource can contain 1 to n LK resources.

    • An archive resource should contain only one SK resource (remember that you can make your own SK resources, which can be very sophisticated subsets).

    • An archive resource can contain one grammar resource.

    You can attach more than one archive to a recognizer, providing that you stay within the rules for the total number of permitted resources.

    Using MkArchive

    Setting MkArchive

    1. Execute the rdkConsole.bat file that is in the rdk folder.

    2. The tool has options that you can set. To see the options, type the help command:

       % MkArchive --help

    3. Follow the instructions.

    Compiling two resource files into one *.res archive file

    1. Execute the rdkConsole.bat file that is in the rdk folder.

    2. Make sure that your resources have a \*.res extension,

    3. In the rdkCompile.bat console, go to the directory required and use this command line:

       % MkArchive -o name_of_new_archive.res name_of_first_resource.res name_of_second_resource.res

    This will create a name_of_new_archive.res file in the same location.

    Digital Text Matcher

    Recognition performance is closely linked to the resources you attach to your recognizer in order to define its data format. It may be helpful to check whether a given string matches one of your resource entries or not. To do this, you can use a DigitalTextMatcher object.

    The DigitalTextMatcher interface creates an engine object in which you can test your resources. To use it, follow these steps:

    • Create a DigitalTextMatcher type of engine object.

    • Attach the resource that you wish to test: you can attach several resources if necessary.

    • Send the string of digital text that you are looking to match using matches(). This returns true or false, depending on whether or not the text can be matched to a resource entry.

    • Destroy the DigitalTextMatcher engine object once you have finished.

    When testing in Asian languages, make sure that you are using a UTF-8 charset.
    Important note for Turkish: as Turkish has unique case-mapping rules for Unicode, to test Turkish linguistic resources, a Turkish Alphabet Knowledge resource must be attached too. If this is not the case, casing cannot be handled correctly.

    Casing schemes

    MyScript Text has casing parameters, set as Vocabulary properties. These properties determine how the casing that is presented in the given resource will be considered during the recognition process. These can be set in a resource’s properties when you build it in MyScript Language Resource Tool.

    • STRICT_CASE
      This applies the casing in the strictest sense, so the entry will only be recognized in exactly the same form as it appears in the resource. For example, if the entry is “aBc”, only “aBc” can be recognized, in that combination of character case.

    • LAX_CASE
      Any casing combination can be recognized, regardless of the casing in which the entries exist in the resource. This type can be used for example, in form processing, with case normalization made in post-processing.

    • LEGACY_CASE
      This applies the casing rules that have always existed in previous versions of MyScript Text (see guidelines below).

    Case variations

    Each resource holds its own casing scheme, defined in its properties. It is also possible to ask the recognizer to ignore case variations in two ways.

    Discarding case variations in the result

    In situations where case is unimportant, turning off case variations avoids redundancy in the top list entries. In the example below, it is hard to distinguish the case of the letters. You will see how discarding the case variations can affect the top list returned in MyScript Text recognition results. For example:

    Ink sample Top list with case variations on Top list with case variations off
    Cut
    Lut
    cut
    lut
    Cat
    Lat
    cat
    lat    		 Cut
    Lut
    Lat
    Cat

    Identical entries with case differences have been ignored. As a consequence, the normalized recognition scores would be affected, as the deviation between results will be slightly higher.

    In your MyScript Text application, use the DISCARD_CASE_VARIATIONS recognizer property to turn the case variations on or off (using a true/false Boolean). Turn the case variations off by setting the DISCARD_CASE_VARIATIONS property to “true”.

    DISABLE_CASING_RULES used to disable the default constraints in earlier versions of MyScript Text is now deprecated.

    Overriding case variations in the recognizer

    There is a recognizer property available that will ignore all the casing schemes specified for all the resources attached to that recognizer. This is the OVERRIDE_CASING_SCHEME property. This is an easy way of ignoring the casing set for a number of resources without having to modify each of them.

    Legacy type casing

    If you apply a LEGACY_CASE type, the rules below apply:

    “LEGACY” rules for single words

    • A word entered in lowercase can also be recognized in Titlecase and UPPERCASE:
    “hello” enables recognition of:
    “hello”
    “Hello”
    “HELLO”
    • A word entered in Titlecase can also be recognized in UPPERCASE:
    “Hello” enables recognition of:
    “Hello”
    “HELLO”
    • A word entered in UPPERCASE can only be recognized in UPPERCASE:
    “HELLO” only enables recognition of:
    “HELLO”
    • A word entered in an unconventional merge of lowercase and UPPERCASE can also be recognized in its original case and* UPPERCASE*:
    “heLLo” enables recognition of:
    “heLLo”
    “HELLO”

    “LEGACY” rules for an expression

    • If you enter an expression in lowercase, you can recognize each word in lowercase, Titlecase and UPPERCASE independently:
    “hello world” enables recognition of the following strings:
    “hello world”
    “hello World”
    “hello WORLD”
    “Hello world”
    “Hello World”
    “Hello WORLD”
    “HELLO world”
    “HELLO World”
    “HELLO WORLD”
    • If you enter an expression in Titlecase, you can recognize each word in Titlecase and UPPERCASE independently:
    “Hello World” enables recognition of the following strings:
    “Hello World”
    “Hello WORLD”
    “HELLO World”
    “HELLO WORLD”
    • If you enter an expression in UPPERCASE, it can only be recognized in UPPERCASE:
    “HELLO WORLD” enables recognition of the following strings:
    “HELLO WORLD”
    • If you enter an expression in an unusual combination of lowercase and UPPERCASE, you can recognize each word if it appears in the given combination of cases, and you can also recognize each word independently in Titlecase and UPPERCASE:
    “HEllo worlD” enables recognition of the following strings:
    “HEllo worlD”
    “HEllo WorlD”
    “HEllo WORLD”
    “HELLO worlD”
    “HELLO WorlD”
    “HELLO WORLD”

    Rules for inter-segment and intra-segment spaces

    The charset provided by MyScript Text offers 2 different “space” characters; this has an effect on the casing:

    • the ASCII character 32 (or codepoint U+20): this is a normal space; this uses the casing rules described above.

    • the ASCII character 160 (or codepoint U+A0): this is a “non-breaking space”; adding this type of space means that the elements it connects must use the same casing.

    Hello<U+20>World can be recognized as: Hello<U+A0>World can be recognized only as:
    Hello World Hello World
    Hello WORLD HELLO WORLD
    HELLO World  
    HELLO WORLD  

    Rules for dashes or hyphens

    The dash character is handled like the non-breaking space described above: Hello-World can only be recognized as:

    • Hello-World or,
    • HELLO-WORLD
    The current version of MyScript Text does not allow for hyphenation over two lines of text.

    Accentuation schemes

    Unlike English which has no diacritics on its letters, many alphabets have letters with various accents. To help solve problems linked with differences in how accented characters are written and how accents may be recognized from a resource, properties have been added that control how strictly accents are applied.

    MyScript Text has accentuation parameters, set as Vocabulary properties. These properties determine how the accents that are presented in the resource will be considered during the recognition process. These can also be set in a resource’s properties when you build it in MyScript Language Resource Tool.

    Accentuation scheme Effects
    STRICT_ACCENTUATION This applies the accents in the strictest sense, so the entry will only be recognized in exactly the same form as it appears in the resource. For example, if the resource entry is Élysées, only Élysées can be recognized, with that exact combination of accented capitals and lowercase letters, and all accents must be present.
    LAX_ACCENTUATION Any accent combination can be recognized, regardless of the accents used in the resource’s entries. Letters can be recognized with or without accents.

    Another remark is that, when strict accentuation is used, you may get a result with a different diacritic but this will be reflected in the score. For example:

    Accentuation scheme Resource entry Written word  
    STRICT_ACCENTUATION thé thê

    In this case, a result of “thê” is impossible, as it is not in the resource. However it is likely for that written word to get a result of “thé”, but the score will only be poor to average.

    Use case

    We recommend using this property in a common situation, when you have a resource that contains uppercase entries with no accents: writers may not write in uppercase and may use accents (e.g. the lexicon contains ALLEE SEGUR and the writer writes “Allée Ségur”). To avoid recognition errors because of casing and accentuation differences, you should use LAX casing and LAX accentuation and then use a post processing method to convert all the results to uppercase, so they match your database entries.

    Example

    The example below shows a case of applying lax accentuation. In this case, capital letters are also involved, so the casing scheme must also be set. In this example, the lexicon contains no accents. But the handwriting sample has some accents included. Thanks to the settings chosen (Lax Accentuation and Lax Casing), the sample is correctly recognized, even though there are differences between the accentuation in the sample and in the lexicon entry. Here, the final result has been converted to uppercase before storage in a database, which in this case removes the accents.

    Accentuation variations

    Each resource holds its own accentuation scheme, defined in its properties. It is also possible to ask the recognizer to ignore accentuation variations in two ways:

    • Discarding accentuation variations in the result
      In situations where accentuation is unimportant, turning off variations avoids redundancy in the top list entries. Discarding the accentuation variations can affect the top list returned in the recognition results. If you discard the accentuation variations, entries that are identical but with accentuation differences will be ignored. As a consequence, the normalized recognition scores would be affected, as the deviation between results will be slightly higher.
      In your MyScript Text application, use the DISCARD_ACCENTUATION_VARIATIONS recognizer property to turn the case variations on or off (using a true/false Boolean). Turn the case variations off by setting the DISCARD_ACCENTUATION_VARIATIONS property to “true”.

    • Overriding accentuation in the recognizer
      You can easily ignore the accentuation schemes specified for all the resources attached to your recognizer having to modify each of them. This is possible thanks to the OVERRIDE_ACCENTUATION_SCHEME property.
