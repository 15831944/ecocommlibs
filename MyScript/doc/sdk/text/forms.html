Form-based recognition

  • SDK
  • Text
  • Form-based recognition

    • Guidelines for application building

    A form is a document in which writers can only write certain things in certain areas. There are various ways in which MyScript Text can help you customize the recognition engine to suit a form-based application.

    With MyScript Text, each field of a form can use different parameters (different writing styles, different alphabet subsets, recognize different language elements etc.). This naturally depends on the content you expect each field to have. This means that the first thing you have to do is to analyze clearly what data your form must collect and the most efficient ways to do this, with a view to handwriting recognition. You need to plan and test your forms before deploying them, to be sure that the way you are collecting data is the most suitable and effective.

    The most important global rule when building a form application is:

    The more constraints imposed on the writer, the higher the recognition effectiveness.

    Obviously these constraints will be experienced by the writers as they fill in the form. Constraints can be applied at three main levels:

    • the handwriting style
      Depending on how the field appears, the writer will almost always use a specific handwriting style. You can use isolated characters or cursive writing style. To use iosolated characters style in a form, you must constrain the writer to enter each character in a box or similar delimited area. This style is the most restrictive one and therefore it ensures excellent recognition accuracy; this does not mean that you should not use any linguistic resources, because the recognizer still needs to use the language expert even if characters are supplied already segmented. Cursive style however, is the most natural one for the writer. There are, however, no constraints on character separation; this is why it is impossible to recognize cursive handwriting without using at least one linguistic knowledge (LK) resource.
      If for some reason, you cannot attach a linguistic knowledge resource to a field, you will need to choose an isolated characters Alphabet Knowledge (AK) resource (the [language+country_prefix]-ak-iso.res file) and set up the field as a boxed field: you will not be able to use a cursive handwriting Alphabet Knowledge Resource without an LK resource.

    • the field layout
      These restrictions refer mainly to two aspects: the level of detail assigned to the field and its size. More specific fields use resources more effectively. These resources are usually better targeted and provide more accurate results. For example, it is much more advantageous to use “Title”, “FirstName”, “LastName”, “MaidenName” and “MiddleName” fields, than to provide only a “Name” field. But you will need to develop more precise resources to match each field. On the other hand, if you have a resource that can recognize several fields (such as a city+zipcode resource), you should use correlated fields during recognition to make the most out of the “multi-field” resource. You can combine fields for recognition purposes and split them at a later moment if necessary (for example for storing field data in a database). An example of this would be a postal code field and a city field (these are provided by default for many languages in MyScript Text):

    Always make sure that you are familiar with what a given resource can recognize and use it accordingly.

    Field size must match the potential size of resource entries. You cannot have a field of 5 characters if your lexicon entries are up to 10 characters long! And the opposite is true: if you need to recognize only a certain amount of characters, adjust the field length accordingly. There is a property for instances of a lexicon that can tell you the maximum word length within that lexicon (MAX_WORD_LENGTH); it may of course be more difficult to predict the length of a potential return from a regular expression-based resource.

    Perform appropriate data collection. You will of course improve your ability to lay out forms correctly by performing data collection to better determine writers’ habits in given circumstances: data collection helps you to see what the input possibilities are on a field, so you can see how your data formats need to be defined. Remember that MyScript’s support services can provide high level expertise in improving recognition accuracy in your solution.

    • the expected field content
      You can also constrain the writer to enter only a certain type of data in a field (for example, a surname, a city name, a date, an ID code, etc.), in a certain order. In your recognizer object, you need to put these constraints in terms that the recognition engine can understand, by defining the syntax, or data format, you expect the field entry to have.

    For each field, there are several possible ways of defining a field’s data format: - you can restrict the character set recognized in that field, - you can use a linguistic resource to restrict recognition to certain words, names, compound words or syntaxes.

    Remember that MyScript Text offers the MyScript Language Resource Tool for defining your own custom resources.

    You can restrict the character set using a Subset Knowledge ( SK) resource. An SK resource defines which characters can be recognized by the MyScript engine. For example, if you know that your field can only contain numerals, you can attach a digits subset resource and the recognition engine will only consider that there are numbers in that field.

    You can use linguistic knowledge resources. The data format of any handwriting context describes the possible characters that a field may contain and the order in which they may appear. For example, a date is usually written in a limited number of ways (1/1/01, 1-1-01, 1.1.01 and so on) and you can cater for this by using a “date” resource that can recognize a date if it follows one of a number of common syntaxes. The recognizer will look for the closest match between the handwritten item and the possibilities defined in the linguistic knowledge resource.

    Defining boxed fields

    When creating form-based applications, you will no doubt define the areas where the writer can make strokes. This page describes how you can send ink corresponding to boxed fields. We do not recommend the use of “combs” for handwriting recognition.

    Boxed fields are used to define a field whose characters you wish to separate from each other. The field contains a series of contiguous guide boxes. This prompts the writer to write each character in a separate box. On a programming level, these guide boxes help you to define the areas in which you will later assign the strokes that the writer has written.

    When performing boxed field recognition, you do the stroke segmentation, because you tell the recognizer at which stroke each character starts. Recognition in boxed fields is done with isolated character recognition.

    The new input model includes a CHAR type of input unit, which you can use for boxed field recognition. To tell the recognizer that you are going to send strokes corresponding to a character inside a box, you have to start a new CHAR type input unit. Then send the strokes. Then stop the input unit. Do this for each character in your boxed field.

    As the writer will also probably see boxes printed on the form, you may want to give the recognizer the coordinates of the guide box that corresponds to the current character (setGuideBox). This helps the recognizer solve geographical discrepancies, such as telling an apostrophe from a comma, or a capital letter from a lowercase one, because of where they are placed within the box’s boundaries.

    Follow this pseudo code for setting up the recognizer:

    create a structured input recognizer
attach any linguistic knowledge resources you require

for each guide box:

start a CHAR input unit
set the guide box characteristics (optional)
set handwriting recommendations (if using several alphabet knowledge resources)
send ink
stop the input unit

ask for recognition
destroy the recognizer

    Anoto User Areas

    In the Anoto SDK, user areas and associated methods are used to retrieve strokes. A user area is a defined, named area on a pattern page, and these are used as form fields in Anoto technology.

    To correctly retrieve strokes with the Anoto SDK, you should use PenStrokes to get the XY arrays. We recommend that you not use Anoto user areas for the purposes of handwriting recognition but retrieve strokes at page level.

    For handwriting recognition, you will achieve increased stroke assignment accuracy by first retrieving strokes for the entire page (at the drawing area level) and then reassigning them to each field, using the stroke assignment methods described earlier.

    Recommended Algorithm

    Based on the solutions we have listed on this page, here is an algorithm that you can use for achieving correct stroke retrieval for handwriting recognition:

    1. get Strokes for the ENTIRE PAGE (drawing area in the pad file) through the Anoto getPenStrokes method.

    2. Define a larger Bounding Box around the field if possible.

    3. Reassign them to each field through the center of gravity calculation

    4. Use the context of the stroke when fields are too close together.

    Using isolated character recognition

    Optimizing isolated character recognition

    Isolated character recognition is vital for the forms industry, as it can provide very low error rates. Because of this, MyScript Text provides a number of features designed to provide more flexible results in isolated character recognition.

    They affect the interaction between the recognizer and the attached linguistic knowledge resources, allowing you to decide how the recognizer should react when it encounters ink samples that do not exactly match the recognizable content described by the LK resources. They can also guarantee that you will obtain some result even if it is an out-of-lexicon result.

    Setting these properties correctly will allow you to implement a robust rejection function, by looking at the new result information, together with the “old school” scores. And without having to tune thresholds on application representative databases.

    An explanatory document is available that gives guidelines for setting these properties. Please read it to discover what settings you should use according to the results you wish to obtain.

    Rules for casing and accentuation in Vocabulary objects such as lexicons have changed. These aspects can be important for good rejection. For information concerning this, please read the help files delivered with the MyScript Language Resource Tool.

    Glyph distortion

    This applies to isolated character recognition only.

    A glyph is a graphic symbol that provides the look or shape for a character. In other words, a glyph provides the visual representation of a character (or the mark you write on paper to represent a certain character). The MyScript engine examines glyphs and tries to determine, by the shapes, loops, size etc, to which digital character they best correspond. Some glyphs are more alike than others and so the engine will choose the most likely match.

    You can set a glyph distortion tolerance. In other words, you can ask the engine to turn down a character candidate that does not have a strong enough probability. Using glyph distortion is a way of setting limits on the potential variability of characters. This is controlled by the GLYPH_DISTORTION property of the recognizer.

    This is a value between 0 and 254. 0 means that very little distortion is allowed: 254 means that maximum distortion is allowed. 255 is a special value (see below).

    For example:

    • A glyph distortion of 0 means that the ink must strongly “look like” a character for it to be recognized (i.e. character probability must be high).

    • A glyph distortion of 254 means that the recognizer will consider characters that do not much resemble the ink sample, as long as they contribute to a result that occurs in the attached linguistic knowledge resources.

    • A glyph distortion of 255 allows the recognizer to consider all of the characters in the attached Alphabet Knowledge resource for a potential match, with priority given to matching a result that occurs in the attached linguistic knowledge resources. At 255, there is no constraint placed. This is the default if no specific level is set.

    Keep in mind that when you set a low glyph distortion allowance, close to 0, you impose a stricter form of decision making on the recognition engine, which means that in some cases you may not get a result but the results you do obtain are more reliable. It also means that the character expert in the MyScript engine may prematurely reject a glyph, and return a replacement character instead (of type U+FFFD), and flag this as “unknown” (although this will occur only in out-of-lexicon candidates).

    An explanatory document is available that gives guidelines for setting these properties. Please read it to discover what settings you should use according to the results you wish to obtain.

    Flags related to isolated character results

    This applies to isolated character recognition only.

    When you use the recognizer properties devoted to optimizing isolated character recognition results, namely the GLYPH_DISTORTION, SPELLING_DISTORTION and ENABLE_OUT_OF_LEXICON properties, special output flags are given in the result. When characters are substituted or considered as omitted or when out of lexicon matches are made, this will be reflected in the score, and you will raise a candidate flag on the candidate.

    Several types of flags exist:

    Flag Description
    SUPERFLUOUS The candidate contains at least one superfluous character.
    SUBSTITUTED The candidate contains at least one substituted character.
    OMITTED The candidate contains at least one omitted character.
    OUT_OF_LEXICON The candidate does not correspond to a lexical match and has been chosen out-of-lexicon.
    UKNOWN The recognizer cannot match this to a known character so a replacement “unknown” character is supplied (Unicode character U+FFFD).
    ERASED The candidate contains text that has been corrected by an erasure phenomenon.
    EMPHASIZED The candidate has been emphasized, in other words, the writer writes over the top of the letter to make it stand out more clearly.

    You also receive other output information, such as the recognition scores and the “spelling distortion ratio” (sdr): this is expressed as a percentage and refers to the volume of characters that have been affected in order to achieve a lexical match (characters that have been manipulated by the distortions).

    The clearest way to see this is in some examples. The settings used are listed for each example:

It inserts “l” and “r” into the result but it does warn you, with an “omitted flag” on each.

    Spelling distortion

    This applies to isolated character recognition only.

    The MyScript recognition engine works best when used with linguistic resources, such as a lexicon. It will attempt to match its recognition candidates with the allowed elements in its linguistic resources. The SPELLING_DISTORTION recognizer property affects the lexical attraction. The recognizer will go so far as to “invent” a character that is missing, or ignore a character that is present, in its quest to absolutely find a lexical match for the ink sample. A flag will appear in the result to warn you when these events have occurred. Use spelling distortion if you want to know when the writers have added or omitted characters in relation to the lexical entries involved.

    This is a value between 0 and 255: the strength of the property depends on the value you define. 0 means that no distortion is allowed, so no characters will be substituted or flagged as omitted: 255 means that maximum distortion is allowed, so substitutions are freely allowed. In practice however, there are 4 “key values” that you should use, 0, 50, 100 and 150 (if you use another value, it will be rounded to the nearest key value).

    The article related to isolated character result flags describes the output flags associated with the spelling distortion property and gives examples. To summarize, the flags associated with the spelling distortion are:

    Flag Description
    SUPERFLUOUS The candidate contains at least one superfluous character.
    SUBSTITUTED The candidate contains at least one substituted character.
    OMITTED The candidate contains at least one omitted character.

    The output information also contains the spelling distortion ratio: it is expressed as a percentage and refers to the volume of characters that have been affected (as a percentage of the lexicon entry) in order to achieve a lexical match (characters that have been manipulated by the distortions).

    Caution: the glyph distortion must be enabled so that the language expert returns a substitution, instead of choosing an improbable candidate from the character expert. This feature is time consuming, and should be used with a small lexicon or with a limited regular expression: this means a regular expression with restrictive syntax which does not include the + or * characters. For instance the date and phone number resources are based on limited expression, whereas the email or number resources are based on very open regular expressions.

    Out of lexicon management

    This applies to isolated character recognition only.

    MyScript recognition engine works best when used with linguistic resources as said previously. However it also has the ability to find an “out of lexicon” match, which does not correspond to the entries present in the lexicon.

    This is now a property of the recognizer that you can enable or disable (ENABLE_OUT_OF_LEXICON). When it is enabled, the recognizer can suggest candidates that are not consistent with the attached Linguistic Knowledge resources, but these candidates will be flagged with an “out-of-lexicon” flag. When it is disabled, the recognizer will unfailingly choose a lexicon match (even if it is extremely unlikely) or will return “no recognition result”.

    When the ENABLE_OUT_OF_LEXICON property is turned on, in the result, two more candidates will appear, at the end of the list, after the other matches. The scores in this part are not comparable with the other scores. Note that, for an “out-of-lexicon” candidate to be the winning candidate, there must be no possible in-lexicon match.

    Because of this, many MyScript Text pre-defined resources have been modified: they no longer contain a preset out-of-lexicon capability. If you use resources with an out-of-lexicon capability in them, you will not get a correct result in isolated character recognition. A second copy of these resources now exists called “extended” (this is visible in the file name), indicating the resource that has a built-in out of lexicon (in other words, extended lexical) capability. You can use this when you wish to retain an out-of-lexicon capability in cursive recognition.

    Managing correction in isolated characters

    For isolated characters (which means recognition carried out with the input unit as “char”), three correction situations can be taken into account by the DigitalInkTagger, and they must be performed as demonstrated below:

    • Insertion of characters immediately after a correction

    The new characters must be placed in the next boxes immediately after the correction. To insert a space between two words, the erroneous ink must be corrected and an empty box left after it, otherwise once the correction is removed, the ink will be joined:

    Here for example, “RIDGE” and “ST” will be properly recognized as 2 separated words.

    • A missing pen lift

    This is when the writer has joined together two characters that should be in separate boxes in a boxed field. In other words, the user has not respected the separation between two character input units by lifting the pen. This works in situations where the input contains several consecutive “char” input units, each with a guide box. Please note a missing pen lift cannot be detected if the input does not include the guide box definition. This incident is not taken into account during recognition. To treat it you can carry out cursive recognition, which may give a satisfactory result, or you may rework the ink. For form processing, the field should be sent for manual validation.

    • An “emphasized character”

    The person is unsure that the letter was written clearly enough (a classic example is the ink in the pen not being visible enough), so they write over the top of the letter to make it stand out more clearly. . This works in situations where the input contains a “char” input unit. In Latin script, the recognition result may be reliable, however not in CJK, although for non-Latin languages, you will still be flagged that a writing incident of an “emphasized character” type has occurred. In either case, we recommend that you send for manual validation. It works only in the case where the same letter is written more boldly. It does not take into account a correction attempt (writing a different character over the original).

    To detect errors in isolated character mode, the input must contain the stroke order information (see stroke_order parameter, of addStroke function).

    An Alphabet Knowledge resource must be attached. The isolated character (ISO) resources also take into account the correction objects.

    Optimizing the accuracy

    The more you narrow the parameters that the recognizer must take into account, the more likely you are to obtain a correct result. Recognizing isolated characters is “simpler” than recognizing cursive writing. In structured documents, such as forms, you must aim to produce fields that are:

    • physically constrained, by using combs, or boxed fields, and providing clear instructions on how to fill them in (in capitals, block letters etc.). Use appropriate field lengths.
    • carefully targeted: a Name field is a poor substitute for a combination of FirstName, MiddleInitial and LastName fields. Separating numerical and text data, for example, a street or building number from the street name, means you can restrict recognition to a certain type of character using a subset knowledge resource.

    Score thresholds and error-rejection curves

    Scores are provided to give you an appreciation of the recognizer object’s confidence in its decisions. Scores are also used to determine thresholds, so you can designate a cut-off point. A cut-off point is useful for customizing your recognition results, deciding what to “keep” and what to “discard”. In other words, this allows you to sort relevant results from irrelevant. Using score thresholds and error-rejection curves is ideal if you are doing automated data processing.

    Rejection

    The goal of rejection is identifying ink samples (also referred to as units) that may be incorrectly recognized and that either need manual validation or another type of post-processing.

    What you choose to reject depends on your application:

    • Rejection at text recognition level
      In some applications, you will want to reject at the highest level possible, namely at text recognition level. In this case you would select only the first candidate at that text level as your recognition result. In other words, the processing will be based on the first candidate scores only.

    • Rejection at character or word level

    In isolated character recognition, you may wish to reject individual characters. You can reject from character or word level. In that case, the rejection process will be based on the candidate scores from that corresponding level.

    Determining rejection thresholds

    To carry out rejection, you need to determine a rejection threshold for the normalized recognition score AND/OR the resemblance score. This threshold is the lowest score value that you are willing to accept from the recognizer. If the score is insufficient, the ink sample will be rejected.

    When you use both the normalized recognition score and the resemblance score, if one of these scores is below its associated threshold, the sample will be rejected.

    Too low or to high rejection threshold will impact your results:

    • High rejection threshold
      A high rejection threshold may give you more chance of a correct result. But the disadvantages of too many rejected samples and nil results may outweigh the advantages of higher accuracy.

    • Low threshold value
      A threshold that is too low will result in incorrect results that are not “caught” by manual checking.

    Error-rejection curves

    Incorrectly recognized samples are accounted in the error rate. Samples that are rejected and not attributed a character category to are accounted in the rejection rate. A way of managing the compromise between error rates and rejection rates is to draw up an error-rejection curve. This is a curve that tells you what percentage of results you need to reject in order to achieve a certain error rate.

    The curve below, which is approximate, is based on an isolated character recognition using the normalized recognition score. The curve would not be the same for a different handwriting context. It would vary according to:

    • the AK resource used,
    • the LK and SK resources used,
    • the nature of the samples (characters, words, text),
    • the data (ink samples).

    Building your rejection curve

    You can generate your own error-rejection curve either using the normalized recognition score OR the resemblance score, or normalized recognition score AND the resemblance score.

    Using the normalized recognition score OR the resemblance score

    • Use an ink sample database with the corresponding labels (digital text). You need at least several hundred samples.
    • For each sample, carry out recognition so you have the recognition result (the first candidate at the text level) with its associated resemblance score or normalized recognition score.
    • Sort these results with the expected labels in ascending order of score.

    • Build the curve using the following algorithm:

        For each (RejectionRate(%) From 0% to 100%, every 1%)
      NumberOfRejectedSamples = NumberOfSamples * RejectionRate(%)
      ErrorRate(%) = NumberOfUnrejectedErrors / NumberOfUnrejectedSamples

    RejectedSamples refers to samples that do not meet the current quality threshold, and so are rejected as unrecognized.

    UnrejectedErrors are samples that return an incorrect text label but are over the quality threshold and are considered as acceptable results.

    Using the normalized recognition score AND the resemblance score

    • Use an ink sample database with the corresponding label (several hundred samples).
    • For each sample, carry out recognition so you have the recognition result (the first candidate at the text level) with its associated resemblance score AND normalized recognition score.
    • Your aim in calculating the score here is to sort once by resemblance score to see the threshold for rejected errors, based on a low resemblance score, then to sort again to retrieve results that have a satisfactory resemblance score but a low normalized recognition score. This will help you to adjust the thresholds to reject more incorrect results.

    • Build the curve using the following algorithm:

        For each (RejectionRate(%) from 0% to 100%, every 1%)
      Initialization: CurveErrorRate [RejectionRate]=100%
      For each (ResemblanceRejectionRate (%) from 0% to 100%, every 1%)
          - Sort in ascending order of resemblance score
          - Sort only the unrejected samples from the initial sort in ascending order of NormalizedRecognitionRejectionRate (by the normalized recognition score)
      For each {NormalizedRecognitionRejectionRate(%) from 0% to 100%, every 1%)
          - RejectionRate = ResemblanceRejectionRate + NormalizedRecognitionRejectionRate
          - NumberOfRejectedSamples = NumberOfSamples * RejectionRate
          - ErrorRate = NumberOfUnrejectedErrors / (NumberOfSamples - NumberOfRejectedSamples)
          - If(ErrorRate < CurveErrorRate[RejectionRate])
          CurveErrorRate[RejectionRate] = ErrorRate

    RejectedSamples refers to samples that do not meet the current quality threshold, and so are rejected as unrecognized.

    UnrejectedErrors are samples that return an incorrect text label but are over the quality threshold and are considered as acceptable results.

    CurveErrorRate is an array that keeps the best configuration (based on the resemblance score or the normalized recognition score) for each rejection rate.
