    • About MyScript InkSearch SDK

    MyScript InkSearch aims at automatically indexing handwritten content in real time, allowing users to search for specific terms and quickly find relevant information. InkSearch is an add-on MyScript Text feature. You can use it only if you have purchased the separate InkSearch extension SDK.

    In brief, MyScript InkSearch supplies:

    API for C, Java, and .NET. InkSearch API. edk/your_programming_language/
    InkSearching This code sample shows a full InkSearch process on structured input, accessing a predefined index, creating a query, creating and using a finder object to carry out the search query in the index and retrieving the result and scores. edk/programming_language/examples/inksearch/

    MyScript InkSearch is based on a two-stage process:

    • the indexing stage
      During this phase, an index of the handwritten source material is prepared.

    • the search stage
      During this stage, queries are used to find specific text within the indexed handwriting.

    MyScript InkSearch has the following features:

    • Input format
      MyScript InkSearch can handle various digital ink contents, such as notes and form text fields.

    • Queries
      MyScript InkSearch SDK enables to perform various types of queries: string-based, vocabulary-based (all email addresses for instance).

    • Boolean operations
      MyScript InkSearch SDK allows queries based on Boolean operations: OR, AND, AND NOT, etc.

    • Scores
      Scores are provided to describe how confident the recognizer is that the search results match the query.

    InkSearch can work with cursive handwriting only, for any language.

    Application blueprint

    Indexing

    Indexing is the pre-processing stage that prepares your handwritten material for searching. You can prepare the index just prior to the search, or at an earlier time. In the case of a StructuredInputIndexer, you can also index incrementally, while the ink is still being supplied. The index will be considered finished when the document is finished.

    1. Create an engine.

    2. Set up an ink search index is to create a StructuredInputIndexer. This is a special type of handwriting context.

    3. Attach at least the appropriate Alphabet Knowledge (AK) resource to this indexer (although you can also attach Linguistic Knowledge (LK) resources if your application requires it).

    4. Create an Index input object for the ink you want to index.
      You will set this input object as the source of the Indexer and use it to add strokes (send ink).

    5. Start indexing with the run method.

    6. When you ask for the result with getResult, it will return an Index object containing the computed data. This index object can be stored on the disk for later retrieval.

    Indexing and recognition can occur simultaneously. This applies only to a combination of StructuredInputIndexer and StructuredInputRecognizer type objects.

    Finding ink

    The second stage is to find handwritten text in the index that you created previously.

    1. Create an engine.

    2. Create a Finder object. This is used to carry out search requests on indexed digital ink samples.

    3. Create a Query object, which defines what you want to search for and which supports binary operators such as AND, OR, AND NOT etc. There are different types of queries, and you should choose the one that is appropriate to your search.

    4. Load and attach the Index object to the Finder object. You can add only one Index object at a time. If you add more, the previously attached ones will be detached.
      Attach the Query object to the Finder object.

    5. Use the run method to start the search process.

    6. You can get the result through the getResult method. This outputs the result in a FindResult object that contains the occurrences of the search text that have been found.

    The search is carried out in the index, but to display the result, your application still needs to have access to the original ink.
    A confidence score is calculated automatically for each occurrence found of the search text. You can choose whether or not to make use of this score for your application purposes.

    Resources

    MyScript InkSearch requires the following resources:

    Alphabet Knowledge
    MyScript InkSearch needs an Alphabet Knowledge (AK) resource.

    Linguistic Knowledge
    You can also attach Linguistic Knowledge (LK) resources if your application requires it). During indexation, we recommend you attach a language_country-lk-text.res resource.

    Result

    Depending on the stage you run, the result of the process differs. After the indexing stage, the result is stored in an Index object. This contains the indexed data. After the search stage, the result is stored in the FindResult object. This object contains the list of occurrences of the search text.

    InkSearch environment

    InkSearch API supplies objects you can use to model your search-and-retrieval application for handwritten material.

    Indexers

    An Indexer object is a special type of handwriting context. You can use it to set up an index of handwritten material for search purposes. There are two Indexer types, that combine different ways of indexing handwritten material using MyScript patented technology. These are:

    • StructuredInputIndexer
      This is the indexing method that provides the best search performance. However it tends to make a larger index and the search speed is slower. It is also the highest performing for out-of-lexicon searching.

    • StructuredInputSlimIndexer
      This is the indexing method that produces the smallest indexes in terms of size. It also gives good speed performance.

    Here is a table that will give you an indication of the difference between these two indexer types, based on their performances:

    Index Size per Page Indexation Speed per Page Search Speed per Page Â 
    Indexer 60 KB 12 seconds 2 milliseconds
    SlimIndexer 10 KB 21 seconds 0.6 milliseconds

    (Measured on a Celeron 2.4GHz with 512MB of RAM, 1 page = ~100 words, Latin script, index file previously loaded in RAM.)

    Search performances depend strongly on the medium on which the indexes are stored. On a disk, network etc, search speed is slower, however on RAM it is considerably faster. To optimize InkSearch speed performance, we recommend loading the indexes in memory, either on application startup or later as a background task.

    You can also tune the compromise between index size and search accuracy, using the INDEX_SIZE_ACCURACY_COMPROMISE. This is a value between 0 and 255. You can set a custom value or choose a pre-defined setting. A value below 128 favors speed and optimizes the index size and over 128, it favors quality, which tends to increase the index size. You can choose MINIMIZE_INDEX_SIZE to set an automatic value of 0 (very small index) or MAXIMIZE_ACCURACY (large index but better accuracy).

    To recover an index, you need to retrieve the result of your indexing process. This is done with the getResult() function from the IIndexationResultProvider interface.

    Queries

    There are different Query objects for extracting certain types of information:

    • StringQuery
      You can create a StringQuery, for retrieving one or more words. This query can use the * and ? wildcards.
      You have access to parameters that allow you to specify whether the query is case sensitive (matchCase), whether it should correctly match accented letters (matchAccent) and whether or not you allow wildcard characters (interpretWildcards). The * and ? wildcards are accepted, with â*â accepting any character any number of times and â?â accepting any character once. For example ârecogn*â will match ârecognitionâ, ârecognizeâ, ârecognizerâ, and ârecognize?â will match ârecognizesâ, ârecognizerâ.

    • VocabularyQuery
      You can also create a VocabularyQuery. This is created from a Vocabulary type resource (for example a lexical unit resource such as an e-mail will help find all the e-mail addresses written in your handwritten notes).

    There are also queries based on Boolean operations:

    • OrQuery
      This class carries out a logical OR operation between two Query objects. The search aims to produce digital ink occurrences matching the content of one of the two Query objects used to create the OrQuery object.

    • AndQuery
      This class carries out a logical AND operation between two Query objects. The search aims to produce digital ink occurrences matching the content of both of the Query objects used to create the AndQuery object.

    • AndNotQuery
      This class carries out a logical AND NOT operation between two Query objects. The search aims to produce digital ink occurrences matching the content of the first of the Query objects used to create the AndQuery object but that do not match the content of the second of the Query objects involved.

    • NearQuery
      This class carries out a logical ANDoperation between two Query objects and aims to find occurrences in close proximity to each other. The search aims to produce occurrences that match the content of both of the Query objects used to create the NearQuery object. It gives higher scores to occurrences that are âgeographicallyâ close to each other (for example on the same line).

    Finder

    The Finder carries out the search in the index you have prepared previously, or just prior to searching.

    In the Finder interface you can set a score threshold to set the search sensitivity. Occurrences whose scores do not achieve the score threshold will not be included in the result.

    Scores

    There are two possible scores to describe how confident the engine is that the search results match your query:

    • Occurrence score
      The occurrence score gives a confidence rating for a given occurrence, to describe how well this result matches the query text. You can obtain the occurrence score through the getOccurences method.

    • Global score
      The global score gives a score for the overall âdocumentâ (this depends on how your ink input is set up - this could be a form, a page etc.) and is the combination of all the occurrence scores. This varies from 0.0 for low probability to 1.0 for high probability. You can get the global score through the getGlobalScore method.
