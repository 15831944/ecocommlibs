    • This section provides you with tips and use case examples for advanced use of MyScript Text. This may be interesting to read, even though it does not seem relevant at first, in order to get an overview of what can be achieved with MyScript Text.

    Ink correction

    MyScript Text can detect and describe certain types of corrections and writing incidents, namely:

    • erasure of text: Crossing out, strikethrough, double-strikethrough, scratch out.
    • ink incidents such as missing pen lifts: Missing pen lifts (cursive style writing in boxed fields) or overwritten characters.
    • insertion of text.

    You can use this feature to detect corrections and writing incidents. This is particularly interesting for form processing , so that corrected fields can be identified and sent for manual validation.

    It can also be used in note-taking applications, where you wish to recognize “clean” handwriting that overlaps written corrections. In such cases, you want to filter out the corrections and recognize only the “clean” handwriting. Both scenarios are described in the following.

    “DigitalInkTagger” API aims at dealing with ink corrections. It analyzes the ink, defines its elements, and produces a list of “tags” that describe corrections and writing incidents.

    Tags

    With DigitalInkTagger, each tag comes with its type and the input range so you can isolate them on the page. Tags are also defined in a hierarchical tree:

    Here are the existing tags:

    Tag Description Example
    TEXT_LINE a line of text. The sequence of TEXT_LINE tags splits the digital ink into a top to bottom sequence of lines of text.  
    TEXT_BLOCK a part of a TEXT_LINE with a sequence of strokes that are contiguous in time.  
    CORRECTION_BLOCK a part of a TEXT_LINE composed of an ERASED_TEXT, an ERASURE and possibly an INSERTED_TEXT.  
    INSERTED_TEXT a new piece of writing that replaces an ERASED_TEXT, written above it.  
    ERASED_TEXT this corresponds to the text that has been corrected by an erasure.  
    ERASURE tags an erasure. It has various sub-types.  
    STRIKETHROUGH erasure with one horizontal line through the portion of text.
    DOUBLE_STRIKETHROUGH erasure with two horizontal lines through the portion of text.
    CROSS_OUT erasure with a cross shaped gesture.
    SCRATCH_OUT erasure with a scribble-out gesture of several close pen strokes.
    MISSING_PEN_LIFT when the writer has not respected a space between two character input units by lifting the pen. (Only when using “char” input units, see below).
    EMPHASIZED_CHARACTER when the writer writes over the top of the letter to make it stand out more clearly. (This does not work if a different character is written and only when using “char” input units, see below).
    TEXT_COLUMN this tag is only used in the detection of vertically written Japanese. It can detect when text is written in a columnar fashion, rather than in horizontal lines.  

    Setting tag detection sensitivity

    Use the setDetectionSensitivity parameter to adjust the tags priority for detection. This parameter lies in the range of 0 to 1. 0 means that the tag will not be detected and 1 means that maximum detection of that tag is enabled (however this is likely to lead to an over-detection and erroneous tagging of legitimate strokes). You can set the value of this parameter for all the existing tags. If you wish to turn off detection of a particular type of tag, set this parameter to 0.

    Finding which resource supplied a result

    In some situations, you may wish to know which LK resource was responsible for supplying a recognition candidate. This could be helpful to classify the recognized object as a phone number, as a city, etc. knowing the resource that allowed the recognition, when you attached several LK resources to the recognizer. The GetSourceLinguisticKnowledgeName method returns the name of the linguistic knoweledge used to produce the recognition result. This works only at word level.

    On archive resources, GetSourceLinguisticKnowledgeName will extract the name of the resource within the archive.

    Optimizing the accuracy

    100% handwriting recognition accuracy is an unrealistic goal (even for humans), as most text written by hand contains a certain amount of unreliable elements, due to human error and carelessness. Because of this, applications including handwriting recognition will necessarily involve a stage of manual validation.

    You can however still achieve excellent accuracy rates providing you follow a few simple guidelines. This section summarizes these guidelines.

    Choosing the correct handwriting style

    The type of handwriting that you want to recognize is the first thing that you will choose, by attaching the corresponding AK resource to your handwriting context. Do not make a hasty decision about this!

    The style you choose will depend on to what extent you can constrain your writers. Fortunately, MyScript Text input model allows you to attach one, two or all AK resources to a structured input recognizer. For an unstructured recognizer, however, you can still attach only one AK type.

    Choosing the appropriate resources

    It is important to choose the right resources for each handwriting context. A set of carefully chosen SK and LK resources will make all the difference. We list some use cases hereby:

    • Restricted types of characters
      If you are certain that your field will contain only certain types of characters (for example, only numerals, only uppercase letters etc.), attach an appropriate SK resource.

    • General text
      If your field will contain “free” text, of the most general type, you may need to use the “text” LK resource.

    • MyScript Lingo for country-specific resources
      MyScript Text offers an impressive range of specially targeted, country-specific linguistic resources to help you in recognizing a wide array of data.

    • Custom resources
      If your field needs to contain a specific type of information, such as names of your company employees, or special ID codes, or similar, you may need to create a custom lexicon. You can also create your own resources based on regular expressions. To do this, read the document concerning custom regular expression resources.

    • Testing your application
      We recommend that you test your application’s recognition accuracy with InkTool, trying a variety of resources to determine the correct data format for each input area.

    Accounting for inappropriate strokes

    The constraints listed above will help reduce errors caused by inappropriate strokes (such as extra commas or periods, text that exceeds field areas etc.). There are also some precautions you can take to ensure that you have correct stroke input. These include:

    • Using appropriate strategies for assigning strokes to fields.

    • If you are unsure whether the strokes can be written in the correct order, you may need to use a cursive recognizer. Remember that if you choose the hand printed style, each character must be neatly finished before the next one is started.

    • MyScript has been trained through a wide variety of handwriting styles from a considerable number of different individuals so it can recognize as many different hands as possible. However, slanted handwriting may cause problems, if it exceeds a slant of more than 30 degrees, which then negatively affects stroke recognition. An exaggerated slant could be the source of recognition difficulties.

    • Make sure that spaces between words are clearly larger than those between letters. This assists the recognizer with its segmentation process.

    Optimizing the speed

    There are two ways in which you can potentially fine-tune the recognition speed:

    • Adjust the recognition speed versus quality (or accuracy) trade-off
      The “speed versus quality compromise” (SPEED_QUALITY_COMPROMISE) property lets you find the best equilibrium between response time (higher speed results in a reduction in quality) and recognition accuracy (higher accuracy uses more CPU). The property uses a value between 0 and 255 and can be set for each recognizer. You may find it useful to change this setting if, after elaboration of a prototype, you find that you would like to adjust your accuracy levels. A compromise value of 136 may give improved speed without noticeably sacrificing accuracy. During your development stage, however, we recommend that you keep the default value given.

    • Adjust the top list size
      The size of the top lists you use for your recognition result has a direct influence on your recognition speed. The bigger the top list, the slower the recognition. You can set the DEFAULT_CHARACTER/WORD/TEXT_LIST_SIZE property to set the list size to your needs: its default size is 2, which allows you to get a normalized recognition score but without slowing down the process too much. If you do not need any top lists, you can set this property to 1 which will increase speed. If speed is not your main concern but you want a larger top list, you can increase the list size.

    Controlling the recognition path bandwidth

    Recognition paths and the path bandwidth

    The recognizer works by analyzing sequences of language elements (which you will have provided in the form of the linguistic resources you have attached, lexicons etc.). This generates a number of recognition paths, which correspond to paths in an oriented graph where transitions are labeled by one character.

    At each step, the current recognition paths are extended by following all the outgoing transitions of the ending nodes. The current number of recognition paths generally grows exponentially. Therefore, at each step, the recognizer has to select a limited number of recognition paths and discard the others. This limit is known as the recognition path bandwidth.

    If we imagine this process as a tree (a lexical tree), each recognition path is a branch on the tree. The longer the ink sample, the more branches the tree will have. (Here there are only 3 characters in the ink sample, a, b and c).

    The tree grows exponentially as more and more possibilities are explored, so it would not be possible to keep absolutely every “branch” of the tree.

    Instead the part of the MyScript engine that deals with the lexical tree (which is in fact the language expert section) keeps only the best N paths at each step. You can control the number of paths that are kept, if you need to.

    Using the recognition path bandwidth

    By default the recognition path bandwidth is shared among the recognition paths generated from all the attached vocabularies (lexicons, lexical unit resources). This can lead to one of the following situations, which can negatively impact the working of the recognizer:

    Example one: Low-frequency vocabulary
    If the frequency is low on the attached vocabulary, such as an out-of-lexicon resource, a problem of prematurely discarded paths may occur. Out-of-lexicon generally has low frequency so that it does not outperform the other resources attached. This means that when the recognizer starts generating paths, it gives priority to the higher frequency resources. These quickly use up all the bandwidth, leaving none left for the out-of-lexicon resource.

    However, if the high frequency resources fail to find a match, the paths will end and recognition will produce a null result because the out-of-lexicon resource’s paths have ended prematurely.

    To address this problem we need to reserve a certain ratio of the recognition path bandwidth to the out-of-lexicon vocabulary so that it can take its turn when all the recognition paths from the other vocabularies stop.

    Example two: Vocabulary with very open recognition
    Another unhelpful situation occurs when a vocabulary resource that defines a rather free syntax with middle range frequencies overpopulates the whole recognition path bandwidth to the detriment of the other vocabularies.

    Let’s use an e-mail resource as an example. This e-mail resource has a standard frequency but it accounts for a very wide spectrum of possible recognition matches (an e-mail address can use just about any character to begin with). This means that it will generate a large number of paths, probably to the exclusion of other resources attached. However, the e-mail resource is really looking for one particular character in order to confirm the presence of an e-mail address and this is the “@” character. If this is not found, recognition stops, and because the bandwidth has been used up prematurely, the result is null.

    To address this problem, we need to limit the amount of bandwidth that this type of vocabulary can use, to make sure that it is not too “greedy” in the amount of bandwidth it consumes.

    How to control the bandwidth

    To solve these issues, properties have been added for Vocabulary objects that control how that object can use the bandwidth.

    One property, RESERVED_RECOGNITION_PATHS is for controlling the minimum amount of bandwidth that can be kept aside for a given resource. This is a ratio, expressed as a value from 0 to 1.

    Another property, RECOGNITION_PATH_LIMIT controls the maximum amount of bandwidth that a given resource is allowed to consume. Again this is a ratio, expressed as a value between 0 and 1. 1 means that the entire bandwidth can be used by that resource if required.

    If you do not set these properties explicitly, they will default to: 0 for RESERVED_RECOGNITION_PATHS and 1 for RECOGNITION_PATH_LIMIT.

    The selected candidate is actually incorrect. Kay Rode2 is obviously not the perfect match to the ink. This is reflected in the mediocre text level resemblance score. The resemblance score for the first part of the text (“Kay”) was better, but the text level score has fallen sharply due to the high possible confusion of the characters in “Rode2/Rodez”.

    Why did the recognizer produce numbers as part of the text? Because for now, we have given no LK to the recognizer. It must rely only the AK that it has at its disposal. If you want the recognition result to make use of handwriting understanding, you need to give it more resources. A human reader will guess that this is a name, and so would not consider putting a number in the result. For the recognizer to make the same assumption, you would need to give it more information about the expected input.

    Recognition when adding the LK

    The example here will demonstrate the effect of adding LK resources.

    To find the final recognition result, all potential matches are tested according to the LK resources made available to the recognition context. These include lexicons or lexical unit resources for word recognition, and grammars and language models for multiple word recognition. For example, you could constrain the alphabet with an SK resource, eliminating letters or numbers if they are not required in a field. Lexicons and lexical unit resources limit recognition to certain words or groups of items. We can compare the difference this can make by using an example.

    Using a subset of characters

    The recognizer has chosen the best individual elements from the characters at its disposal: letters, numbers, symbols. Observe below the result when you limit the set of recognizable characters to just letters:

    Ink sample Text level result - only upper and lower case letters, no linguistic knowledge
    Kayf. Rodez

    You can see there is an improvement. There are now no numbers in the result, but there are still errors and the text layout is a little strange.

    Using a full name data format

    However the best result would be achieved by examining the probable input and deciding what ink you are expecting to receive. In this case, it is clear that you need to recognize a full name. If we were using a full name data format, that recognizes a combination of first names, middle initials (in upper case) and surnames drawn from specialized lexicons, any nonsensical elements would be eliminated.

    Ink sample Text level result - only letters, using full name data format
    Kay S. Rodez

    By a process of elimination, the unsuitable top list entries will be discarded, until the result with the highest score at the highest recognition level is found to match with the linguistic resources available. This is how MyScript can eliminate character ambiguity through the use of linguistic resources.

    Displaying a partial result

    For certain applications such as input methods, it may be useful to be able to display a partial result. This is especially the case when you use incremental recognition. In general, the recognizer will try and return a result that most closely corresponds to the ink strokes written so far. However those ink strokes may correspond to the start of a longer word still in progress. In such a case, you may wish to obtain results derived from the starting characters of other, longer items in the attached LK resource.

    In general, MyScript Text would not supply this behavior. It would favor whole items from the LK resource. To take this type of case into account, you can use the GetResultEx method. It enables the extension of the GetResult method, usually used to call for the recognition result. If you use this function instead of the standard GetResult, some candidate objects may be flagged with a VO_CANDIDATE_FLAG_PARTIAL flag. This indicates candidate objects that are potentially correct because they match the initial characters of other items supported by the LK resource, even if they themselves are not supported.

    For example:

    You are using a lexicon that contains these words:

    boarder
boathouse
boo

    And your ink sample is the following:

    Using GetResult, MyScript Text would return only “boo” as a candidate. Indeed, this best fits the current ink sample and the word “boa” is not in the lexicon. By using GetResultEx however, it would also return “boa”. This is because “boa” matches the initial characters of the entries “boarder” and “boathouse” from the lexicon. “boa” would be flagged in the iterator as a partial result, however, because it is not in the lexicon as a whole word.

    The application can display several candidate objects. This reflects more faithfully what the writer has written so far. It also takes into account what they may intend to write.

    Recognizing circled characters

    Circled characters (i.e. one or more characters enclosed within a circle) can be recognized with MyScript Text in cursive style. A circled character, or a circled word, is often associated with a user event, such as action commands. For example, to recognize text and send it to an SMS, you could use an “S” circled character and associate a behavior with it. Characters have to be circled in one stroke, once the character is written. MyScript Text cannot recognize circled characters when the circle is written first.

    Circled character integration

    Special MyScript internal characters have been created to take into account circled characters: these are the OPEN_SURROUNDING and CLOSE_SURROUNDING characters. The “open” character symbolizes the beginning of the circled string and the “close” character indicates the end.

    To recognize circled items, you must define them in the LK resource that you are attaching to the recognizer. This can be a lexicon or a lexical unit resource. Define each item that you need to recognize. This can be just a letter or short words. In your resource, define the entry like this:

    Open_Surrounding AB Close_Surrounding

    This will recognize for example the circled string AB.

    Circled character recognition is supported only in the cursive recognizer but not for isolated characters.

    Encoding

    MyScript uses the Unicode private use area to encode the “open” and the “close” characters:

    • For OPEN_SURROUNDING, the code point U+F0000 is used.

    • For VO_CLOSE_SURROUNDING, the code point U+F0001 is used.

    As a reminder, the following table shows the corresponding encodings in several charsets for these two code points:

    Name Code point Java/.NET C/C++ UTF-8 encoding UTF-16 encoding
    VO_OPEN_SURROUNDING U+F0000 “\uDB80\uDC00” U”\uDB80\uDC00” (f3, b0, 80, 80) (db80, dc00)
    VO_CLOSE_SURROUNDING U+F0001 “\uDB80\uDC01” U”\uDB80\uDC01” (f3, b0, 80, 81) (db80, dc01)

    Vertically-written Japanese

    Vertically-written Japanese and English

    It is possible to support Japanese characters written in a vertical style
    (traditional tategaki format) with MyScript Lingo Japanese
    language pack. You can use the resource desgined for recognizing
    vertical characters: ja_JP-ak-vertical.res. This resource covers the
    JISX0213 standard, which corresponds to the kanji from JISX0213 level 1 and
    level 2. This resource also supports mix of vertical Japanese and English words written vertically, as shown in the next figure. English written vertically can be recognized provided that it is writtent one letter beneath the other.

    Mixed horizontal and vertical writing

    Writers of Japanese may write either horizontally, Western-style, or vertically, particularly in a school setting or for formal documents. If you are unsure
    whether users of your application will be writing horizontally or vertically,
    you will need to account for this in your application by running the
    DigitalInkTagger before running handwriting recognition. The
    DigitalInkTagger can return two tags, TEXT_LINE and TEXT_COLUMN: A
    TEXT_LINE tag indicates that the writing is horizontal: in that case you
    should attach the standard ja_JP-ak-cur.res AK resource. A TEXT_COLUMN tag
    indicates that the writing is vertical: in that case you should attach the
    ja_JP-ak-vertical.res AK resource.
