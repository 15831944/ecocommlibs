    • About MyScript Shape SDK

    MyScript Shape SDK provides you with the API based on MyScript Shape technology, enabling the recognition of basic geometric shapes and the beautification of shapes presentation and layout.

    In brief MyScript Shape supplies:

    Item Description  Location
    API for C, Java, and .NET. Shape API edk/your_programming_language/
    MyScriptShape The specific library engine/
    Resource files These files contain specific data needed to perform shape recognition. resources/
    SingleShapeRecognition, MultipleShapesRecognitionBeautification, IncrementalShapeRecognitionBeautification, ScratchoutDetection These code samples demonstrate recognition and beautification of a shape document containing only one shape, containing several shapes, how to add more ink to an existing drawing, how to scratch out edk/your_programming_language/example/shape/
    AdjustingModelsDetectionSensitivity In this example, a shape is drawn, which is hard to distinguish between an ellipse and a rectangle. By disabling the detection sensitivity on elliptical models (which stops them being “recognizable”), the shape is correctly recognized as a rectangle. edk/your_programming_language/example/shape/
    AdjustingBeautifierProperties This example describes a process whereby recognition and beautification are carried out, then a change is made to beautifier properties (such as the AlignmentStrength) and beautification is run again, to display the effects of changing the property settings. edk/your_programming_language/example/shape/

    Application blueprint

    The figure below describes the MyScript Shape SDK workflow.

    A shape recognition with MyScript Shape SDK requires that:

    1. you create a Shape recognizer, once the engine object, enabling MyScript environment, is set. The Shape recognizer will recognize a shape matching a group of strokes.

    2. you attach the resource file (Shape Knowledge in the scheme above) to the recognizer AND to the beautifier. The data laid down in the resource files inform the recognizer on the forms of the existing shapes.

    3. you send the strokes that your application has gathered from user handwriting with an electronic pen or a smartphone, tablet or an interactive whiteboard.

    4. you start the recognition up. The recognizer will find one or more primitives (line segment or elliptic arc): these are the basic geometric components that make up each shape. It will then analyze the set of primitives to predict which type of shape is the best match. For example, 3 line primitives could be a triangle, or an open polyline, but could not be a circle.

    5. you can run a beautification. Following recognition, you can perform a beautification process, to equalize shape sizes, extremities, alignment and rotation according to the provided constraints. For example, an application for strict diagrams will require more constraints (so, higher beautification) than a sketching application.

    6. you get back the recognition result in the form of a list.

    Environment

    This subsection is an overview of the main objects of MyScript Shape environment. APIs are described later.

    Shape recognizer

    The shapeRecognizer is specialized for shape recognition. The recognizer will process the input digital ink as a series of strokes.

    Shape beautifier

    The shapeBeautifier performs the shape beautification.

    Shape knowledge

    This special resource file is used for recognition of shapes, providing “shape knowledge” to the shapeBeautifier and to the recognizer so it is capable of recognizing shapes or lines.

    Segments

    The recognizer considers groups of strokes. These groups are referred to as segments.

    Primitives

    A primitive is a basic geometric form component. A primitive can be a line, a decorated line, an elliptic arc or a decorated elliptic arc.

    Recognition result

    The shape recognition result is a list of recognized shape candidates for each group of strokes, as partitioned by the recognizer. As explained, the recognizer analyzes the segments and outputs a list of matching shape candidates for each segment. The result is thus returned in the form of a list of recognition items per segment. This resulting list is stored in a shapeDocument object.

    Running a recognition

    A shape recognition with MyScript Shape SDK requires various steps hereby described:

    1. Create a Shape recognizer, once the engine object, enabling MyScript environment, is set. use the method createShapeRecognizer to do so.

    2. Load and attach the resource file (Shape Knowledge in the scheme above) to the recognizer AND to the beautifier.

    3. Send the strokes that your application has gathered into the shapeDocument. Use the method addStroketo do so.

    4. Start the recognition using the method process.

    5. Run a beautification using the method process.

    6. Get the recognition result in the form of a list using the method getSegmentAt. Learn how to parse this result here.

    Check the Tutorial section or the code samples provided with MyScript Shape SDK (in the edk/[programming_language_name]/examples folder) to know in details how to implement the whole Shape recognition process.

    Parsing the result

    Steps

    After the recognition process, to explore the resulting list, you will need to:

    1. Get the number of segments
      The resulting list is stored in a shapeDocument object. This object contains the segments. You obtain the number of segments after the recognition process thanks to the method getSegmentCount (from the shapeDocument interface).

    2. Get the segments
      You will use the methods getSegmentAt, if there are any, from the shapeDocument interface, to get one segment contained in the shapeDocument.

    3. Go across this segment
      From the segment interface, you can use getCandidateCount, getCandidateAt methods (if there is any recognized shape in this segment) to reach the recognized shapeCandidate objects.

    4. Get the information on the recognized shape
      Once you get one shape candidate, you can reach the details on this shape thanks to the methods:

      • getModel, and then getLabel to get the name of the recognized shape
      • getResemblanceScore, getNormalizedRecognitionScore to get the scores
      • getPrimitiveCount, getPrimitiveAt to reach the data on the primitives forming the shape candidate

    Using the beautification

    You can enhance the neatness and control the alignment of recognized shapes through the beautification process.

    Processing

    The beautification process consists in modifying the coordinates of the recognized shapes, after the recognition process, according to geometric constraints:

    1. The beautifier takes the shapeCandidate objects from the shape document as input. This contains the data on the primitives composing the shapeCandidate objects: the coordinates of the primitives.

    2. The beautifier resolves geometric constraints such as plausible connections, alignment and congruence. Then, it consequently modifies the coordinates of the primitives forming the shape candidates, for each segment.

    Constraints

    There are three geometric constraints that are the alignment, the connection and the congruence (see the illustration above).

    You can adjust these constraints by properly setting the corresponding parameters. Your settings depend on the requirements of your application. For example a free sketching mode will require much looser constraints than a flow chart mode where more evenly formed and aligned shapes are needed. Remember that there needs to be a balance between shape appearances that are aesthetically pleasing, and the original input. Too much beautification may result in shapes that are too far from the writer’s original intent.

    Alignment

    The alignment aims at arranging shapes so that they are positioned in a straight vertical or horizontal line.

    Set the AlignmentStrength property for controlling the alignment. This controls how far shapes need to be from a vertical or horizontal orientation. Setting this at a high value means that you will align shapes that are quite badly misaligned. A low value means that shapes that are highly misaligned will not be modified.

    Connection

    The connection aims at joining shapes together, adjusting the ends of lines to fit shape boundaries.

    Set the ConnectionStrength property for constraining the connection. This property controls how close shapes need to be. A high value means that you will connect even shapes that are not so close together. A low value means that you decide to carry out a connection when shapes are much closer.

    Congruence

    The congruence aims at making size, shape and distribution uniform, given all shapes involved.

    Set the CongruenceStrength property for controlling the congruence. This constrains the variation there needs to be in shape size and distribution. For example, this property would make the shapes in a diagram a more even size and shape, with a more equal distribution. With a high value, you will favor the rescaling and/or alignment of shapes, even when they are quite different in size and alignment. With a low value you will avoid rescaling and/or alignment unless the shapes are very similar in size or alignment. Congruence only has an effect on shapes or lines that have already been aligned by the alignment beautification. These two parameters work in conjunction.

    Forcing the stroke grouping

    The recognizer automatically groups the input strokes into segments. It chooses the combination of stroke candidates that maximizes the global recognition score. It recognizes these segments, and returns all the shapes that have been recognized. This is its default behavior.

    However, you might prefer to clearly specify the strokes that belong to the same group. For instance, it can be useful to bypass the internal recognizer segmentation process, when you are sure that you need to recognize one single shape only. This is possible by forcing the stroke grouping. To force the stroke grouping, you will:

    1. explicitly use the method startSegment() on the shapeDocument,

    2. send the strokes and,

    3. explicitly use endSegment() on the shapeDocument. This is illustrated below.

    In brief:

    In the default mode
    You will be able to recognize multiple shapes only once. The result of the recognition will consist of the typical list of candidates for each recognized shape.

    When you force the segmentation
    You assume that all the input strokes belong to the same shape. You will thus obtain one list of candidates for this single shape.

    Correcting, undoing/redoing

    Scratching out

    MyScript Shape enables scratching out for erasing strokes or shapes.

    The scratch out feature enables to erase several close strokes with a scribble-out gesture, as illustrated on the right.

    ScratchOutDetectionSensitivity controls the scratch out detection sensitivity. This determines how the engine differentiates between real shapes and scratchouts. You can adjust the scratch out feature by modifying the value of ScratchOutDetectionSensitivity. 0.0f disables detection of scratch outs. 1.0f means maximum supported sensitivity (so it will be more inclined to label strokes as a scratch out, in case of doubt).

    Undoing/redoing

    The ShapeHistoryManager object can be used to undo/redo the last strokes added to a shapeDocument. There are various methods from its interface to do so:

    Method Description
    undoLastStroke() removes the last drawn stroke from the shapeDocument. Then, it updates the shapeDocument, as if the last stroke never appeared.
    redoLastStroke() performs the opposite action. It reverts the document back to its state, as it was before the last call to undoLastStroke().
    deleteComponentAt removes the specified stoke or primitive from the shapeDocument. There is no new recognition process performed through this call.
    undeleteComponentAt performs the opposite action. Undeletes the specified stroke or primitive from the shapeDocument.

    There is a code example demonstrating this functionality.

    Shape elements

    The recognizer considers groups of strokes. These groups are referred to as segments. The recognizer analyzes the segments and outputs a list of matching shape candidates for each segment. The result is thus returned in the form of a list of recognition items per segment. This resulting list is stored in a shapeDocument object.

    Segment

    A segment is a group of strokes, as formed by the recognizer. It contains the list of shape candidates (referred to as shapeCandidate on the APIs) that match this segment. The interface of the segment provides different methods:

    Method Description
    getInkRangeCount returns the number of ink ranges that this segment covers
    getInkRangeAt returns the ink range at the specified index
    getCandidateCount returns the number of candidates contained in this segment
    getCandidateAt returns the shape candidate at the specified index
    getSelectedCandidateIndex returns the index of the selected candidate in the segment’s candidate list
    selectCandidateAt select the candidate at the specified index in the segment’s candidate list

    shapeCandidate

    A shapeCandidate stores the recognition information regarding a shape candidate that matches a segment. From its interface, you can access:

    • the label
      This is reachable from the the model of the shapeCandidate. For instance, the label can be triangle, square, circle, etc.

    • parameters
      These are reachable from the primitives associated with the shapeCandidate. For instance, these can be point coordinates for polygons, the center and radius for circles etc.

    • the resemblance score
      This score measures how closely this shape candidate matches the actual ink provided. It is calculated on a scale of 0 to 1, with 1 being the highest level of confidence.

    • the normalized recognition score
      This score indicates the possible confusion that may exist between the different candidates in the top list. It tells how likely the shape candidate is to be correct, in comparison with the other shapes considered. This is a value between 0 and 1. A normalized recognition score 1 indicates stronger likelihood.

    Method Description
    getModel returns the model associated with this shapeCandidate
    getResemblanceScore returns the resemblance score of this shapeCandidate
    getNormalizedRecognitionScore returns the normalized recognition score of this shapeCandidate
    getPrimitiveCount returns the number of primitives this shapeCandidate is made of
    getPrimitiveAt returns the primitive at the specified index

    Primitive

    The primitive is a basic geometric form component. A shapeCandidate is composed of one or more primitives. There are various types of primitive (line, decoratedLine, ellipticArc, decoratedEllipticArc), each of which owns its specific list of APIs.

    All the primitives store specific data (these are the parameters for example point coordinates for a line, center for an ellipticArc).

    The interface of the primitive provides the following methods:

    Method Description
    getData returns the data associated with this primitive
    getPoints returns points generated from the primitive
    getInkRangeCount returns the number of ink ranges that this primitive covers
    getInkRangeAt returns the ink range at the specified index
    You can access ink ranges of primitives using getInkRangeAt/Count methods only before having committed the recognition result. When a recognition result has been committed, the ink ranges of primitives are no longer reachable (see warning note in section Tips).

    Model

    The model stores the label of the shape. For example, this can be arc of circle, ellipse, polyline, etc. The interface of the model provides the following method:

    Method Description
    getLabel returns the label of this model

    Advanced use

    Rejecting candidates

    You can determine how easily recognized shapes will be included in the top list of candidates. To do so, set the value of the RejectDetectionSensitivity parameter. The higher this value, the more shapes need to be very “well drawn” before they are suggested as a recognition candidate. A value of 0.0f means that there are no restrictions on the resemblance scores and that all candidates will be included in the top list (in accordance with the specified list size). 1.0f means only candidates with a very high resemblance score will be included. If there is no shape candidate with a resemblance score high enough, then the candidate list will only contain a ‘reject’ candidate. For most uses, the default value of 0.5 will be sufficient.

    Excluding shapes

    MyScript Shape SDK supports the recognition of various geometric forms. However, if you want to exclude a particular shape, you can set the parameter setModelDetectionSensitivity to a value of 0 on that shape model.

    Building an interactive application

    You can use MyScript Shape SDK in two main use cases:

    The interactive use case
    We refer to this use case as interactive when the application requires strokes to be added intermittently, for example a drawing application on interactive whiteboard. The user can draw, and draw again, adding or deleting parts of a drawing, and receive updated display of the results of recognition and beautification.

    The non-interactive use case
    We refer this use case as non-interactive when the application requires a “finished” document of shapes and drawings to be sent in a single action.

    Implementing the interactive process

    Here, we list the steps for implementing the interactive shape recognition:

    1. Initialize the environment
      1.2. Create an engine
      1.3. Create a shape recognizer
      1.4. Create a beautifier (if needed)
      1.5. Create a shapeDocument

    2. Load and attach resources
      2.1. Load the shape knowledge resource
      2.2. Attach the shape knowledge resource to recognizer
      2.3. Attach shape knowledge resource to beautifier

    3. Start a while loop
      3.1. Send new strokes to shape document
      3.2. Call recognizer
      3.3. Call beautifier (if used)
      3.4. Read segments from the shapeDocument
      3.5. For each shape segment, read shape candidates
      3.6. For each shape candidate, read parameters: ink-range and primitives data (if it is a shape recognized)
      3.7. Update display with recognition results 3.8. End while

    4. Clean-up
      4.1 Detach shape knowledge resource from recognizer
      4.2 Detach shape knowledge resource from beautifier
      4.3. Release shape knowledge resource
      4.4. Release shapeDocument
      4.5. Release recognizer
      4.6. Release beautifier
      4.7. Destroy engine

    Tips for application designers

    Update the display

    It is very important to display updated recognized shapes in interactive shape recognition applications. You will understand why by reading this section.

    In a incremental drawing application, while the user is drawing,

    1. Strokes are captured and displayed by the application.

    2. A timeout is applied to the user’s inactivity, which decides when strokes are sent to the recognizer.

    3. Recognition and beautification processes are called (beautification is optional, though).

    4. Shapes are returned from the recognizer (or the beautifier).

    5. And importantly, display is updated so the users see the results of their drawing.

    Imagine that the user will modify the drawing, adding new strokes or taking some away with a scratchout gesture. This new drawing occurs on a display of the recognized shapes, so the modifications are clearly apparent.

    In an application correctly updating the display, whenever new recognition results are available, the following workflow is performed:

    1. Strokes are captured and displayed.

    2. Again, the timeout is applied to the user’s inactivity, to determine when the new strokes are sent to the recognizer.

    3. Recognition and beautification are called.

    4. Shapes are returned from the recognizer.

    5. And again, display is updated with the modifications made to the drawing.






    Why it is important to update the display:

    It is very important to display the updated result to avoid the situation illustrated by the following figure. This shows that the recognized shape are not visible to the user because the application did not correctly update the recognition result. Only user strokes are displayed. The user thus locates the scratch out according to the reference points that are available: the user strokes, in blue. Consequently, when this scratch out is sent to the recognizer, it is interpreted as a free curve, not as the expected scratch out.

    This can be avoided by using the AUTO_COMMIT_RESULT property. This property indicates whether the recognition result is automatically validated by MyScript internal mechanisms, or whether you prefer to specify when and what to validate and commit. By default, AUTO_COMMIT_RESULT is set to true, which means that the strokes sent by the users, when recognized, are automatically considered as valid recognition result. Set AUTO_COMMIT_RESULT to false if you prefer controlling when and which ink ranges to validate and to commit. Then you will have to use the method commitResult to validate and commit the recognized primitive you like.
    Allow candidate selection

    You might like to allow candidate selection, making the user of your application free to adjust the recognized shapes.

    In an application fitting such feature, while the user is drawing, the following steps are carried out:

    1. Strokes are captured and displayed,

    2. A timeout is applied to the user’s inactivity, which decides when strokes are sent to the SDK.

    3. Recognition and beautification are called.

    4. Shapes are returned from the SDK,

    5. And importantly, display is updated so the users see the results of their drawing.

    Now, we suppose that the user has the possibility to decide that one of the shapes recognized is unsuitable and can choose another shape in its place.

    1. User makes the new shape choice via the interface,

    2. A selectCandidate call is made to the SDK (this is explicit recognition, so no new call is made to the recognizer).

    3. Beautification is called (to “redraw” the shape).

    4. Shapes are returned from the SDK,

    5. And again, display is updated with the new shape included in the drawing.

    MyScript Analyzer extension

    The Analyzer extension delivers MyScript Analyzer technology. The Analyzer can identify different text layouts and text within geometric shapes. It is possible to extract each classified element. Check the section dedicated to Analyzer to learn how to use it with MyScript Shape SDK.
